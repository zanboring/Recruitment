## 摘要

招聘市场每天在BOSS直聘、智联招聘、前程无忧、猎聘、拉勾网、牛客网、应届生求职网、领英等平台上面向公众释放数以千万计的岗位信息。这些信息横跨岗位名称、薪资区间、学历门槛、经验年限、技能标签、公司属性、所在城市等十余个维度，蕴含着丰富的人才供需趋势和市场动向。然而问题在于：数据被割裂在各自平台内部，格式五花八门，缺乏统一标准；企业想了解竞争对手的薪资策略却要逐站手动检索，求职者想了解某类岗位的真实市场价位也往往无从下手；传统的Excel统计方式在面对百万级数据时既慢又不准。针对这些实际困难，我在本科毕业设计阶段独立完成了一套AI招聘数据可视化系统的设计与实现工作。

整套系统的处理管线按照"采集→清洗→入库→统计→展示→智能分析"的顺序串联。在后端，选用Java 22搭配Spring Boot 3.2.5搭建RESTful服务，用MyBatis 3.0.4处理数据库映射，PageHelper 2.1.0解决分页问题；数据采集层面，以WebMagic框架调度任务、Jsoup解析DOM结构，覆盖了上述8个招聘平台，并通过预置10条浏览器User-Agent随机轮换、12至20秒的随机请求间隔、最多4次的指数退避重试等策略来应对平台的反爬机制；数据清洗环节使用正则表达式和字符串标准化，将"15K~25K/月"这类非结构化文本转换为min_salary和max_salary两个字段分别存储15000和25000，学历字段如"本科及以上""统招本科"统一映射为"本科"，并用SHA-256对岗位关键词组做哈希生成43字符的Base64URL编码唯一键job_key来去重；数据持久化选择MySQL 8.0.37，连接池HikariCP配置了最大连接数20、最小空闲连接5；前端工程基于Vue 3.4.0和TypeScript 5.4.0，界面组件来自Element Plus 2.7.0，统计图表用ECharts 5.5.0渲染，状态管理交由Pinia 2.1.7，路由控制使用Vue Router 4.3.0，HTTP通信靠Axios 1.7.0；在AI能力方面，对接了智谱AI的GLM-4模型API，后端用原生的HttpURLConnection而非SDK发起POST请求，通过SSE（Server-Sent Events）协议将模型输出逐Token推送给浏览器端，前端用fetch API的ReadableStream逐块接收，配合marked.js做Markdown渲染，实现了类似ChatGPT的打字机效果，SseEmitter的超时阈值设为120秒。此外，系统还内置了基于权重计算的岗位推荐算法——技能匹配占70%、学历匹配占20%、经验匹配占10%——和基于数据库统计均值并叠加城市/学历/经验/技能因子的薪资预测模型。

系统在功能上拆分为六大模块：数据爬取管理、招聘数据管理、多维度可视化分析、AI智能服务、用户与权限管理、系统日志。其中CrawlServiceImpl.java作为核心爬虫服务实现类，代码量达到1425行，内部使用FixedThreadPool维护2个线程并行抓取，每个平台单次采集上限设为200条（SITE_LIMIT=200）。可视化分析模块的后端统计接口通过一条巧妙的SQL——借助SUBSTRING_INDEX函数配合内联数字表（1到5）——实现了对逗号分隔的技能标签字段的拆分统计，不需要额外的应用层遍历。权限控制层面采用Spring Security配合自定义JwtUtil，密码使用SHA-256加盐哈希存储，Token有效期24小时，ADMIN角色拥有6项权限而USER角色仅有ROLE_USER。

整个开发过程在Cursor编辑器中完成，后端构建工具为Maven，前端用Vite。经过功能测试和性能测试验证，系统页面加载时间在2秒以内，查询接口响应约300毫秒，图表渲染不超过1秒，AI问答首字响应约1.5秒，50个并发用户访问时无超时或报错现象。

**关键词：** 前后端分离架构；Spring Boot；Vue 3；招聘信息采集；ECharts可视化；WebMagic爬虫；智谱AI GLM-4

## ABSTRACT

Major recruitment platforms in China — including BOSS Zhipin, Zhaopin, 51job, Liepin, Lagou, Nowcoder, Yingjiesheng and LinkedIn China — collectively publish tens of millions of job postings each day. These postings span over a dozen dimensions such as job title, salary range, education requirement, experience level, skill tags, company profile, and geographic location, carrying rich intelligence about talent supply-demand dynamics and market trends. However, the data remains siloed across platforms with inconsistent formats and no unified standard. Enterprises seeking to benchmark competitor compensation strategies and job seekers trying to gauge realistic market salaries for specific roles both face significant friction. Traditional spreadsheet-based analysis buckles under the volume and velocity of this information. To address these practical challenges, this thesis presents the design and implementation of an AI-Powered Recruitment Data Visualization System, completed as an undergraduate graduation project.

The system's data pipeline follows a sequential flow: collection → cleaning → storage → aggregation → visualization → intelligent analysis. On the backend, Java 22 and Spring Boot 3.2.5 power the RESTful service layer, while MyBatis 3.0.4 handles object-relational mapping and PageHelper 2.1.0 manages pagination. For data acquisition, WebMagic orchestrates crawl scheduling and Jsoup performs DOM parsing across all eight platforms. Anti-scraping countermeasures include rotating through 10 pre-configured browser User-Agent strings, introducing random delays between 12 and 20 seconds per request, and implementing up to 4 retries with exponential backoff. The cleaning pipeline employs regex-based parsing to convert freeform salary text (e.g., "15K~25K/month") into structured min_salary/max_salary integer fields, normalizes education and experience descriptors into canonical forms, and uses SHA-256 hashing to produce a 43-character Base64URL-encoded job_key for deduplication. MySQL 8.0.37 serves as the persistent store with HikariCP connection pooling (maxPoolSize=20, minIdle=5). The frontend is built on Vue 3.4.0 with TypeScript 5.4.0, utilizing Element Plus 2.7.0 for UI components, ECharts 5.5.0 for chart rendering, Pinia 2.1.7 for state management, Vue Router 4.3.0 for navigation, and Axios 1.7.0 for HTTP communication. AI capabilities are powered by ZhipuAI's GLM-4 model, accessed via native HttpURLConnection (not SDK), with responses streamed to the browser through SSE (Server-Sent Events). The frontend consumes this stream using the Fetch API's ReadableStream and renders Markdown via marked.js, achieving a typewriter-style output effect with a 120-second SseEmitter timeout. Additionally, the system implements a weighted job recommendation algorithm (skill match 70%, education match 20%, experience match 10%) and a salary prediction model based on database aggregate statistics adjusted by city, education, experience, and skill factors.

The system comprises six functional modules: crawl management, recruitment data management, multi-dimensional visualization, AI intelligent services, user authentication and authorization, and system logging. The core crawler service implementation (CrawlServiceImpl.java) spans 1,425 lines, utilizing a FixedThreadPool with 2 threads and a per-platform cap of 200 items per crawl (SITE_LIMIT=200). The visualization backend employs a clever SQL technique using SUBSTRING_INDEX with an inline number table (1 through 5) to split and count comma-separated skill tags without application-level iteration. Access control is enforced through Spring Security with custom JWT handling — passwords are stored as SHA-256 hashes with per-user salts, tokens expire after 24 hours, and the ADMIN role holds 6 authorities versus ROLE_USER for regular users.

Development was conducted in the Cursor editor, with Maven for backend builds and Vite for frontend bundling. Functional and performance testing confirmed page load times under 2 seconds, query responses around 300ms, chart rendering within 1 second, AI first-token latency of approximately 1.5 seconds, and stable operation under 50 concurrent users with no timeouts or errors.

**Key words:** Front-end and Back-end Separation; Spring Boot; Vue 3; Recruitment Data Collection; ECharts Visualization; WebMagic Crawler; ZhipuAI GLM-4

## 1 绪论

### 1.1 选题背景

打开任何一个招聘APP，用户第一眼看到的往往是一个搜索框加上一堆职位卡片。这种模式运行了很多年，基本满足了"企业发职位、求职者投简历"的原始需求。但站在更宏观的视角来看，招聘行业正在经历一场从"信息发布"到"数据决策"的迁移。

BOSS直聘2024年财报显示其日均活跃职位超过1500万个，智联招聘和前程无忧各自也维持着数百万的在线岗位量。这些岗位数据散落在不同平台上，薪资的写法千奇百怪——有的写"15K-25K"，有的写"15000-25000元/月"，还有的干脆写"薪资面议"。学历要求同样花样繁多："本科及以上""统招本科""大学本科""Bachelor"，本质上指的是同一回事。再比如经验年限，"1-3年""一年到三年""1~3yr"说的是同一个区间。如果有人想把某个城市某个岗位类型的平均薪资算出来，手动操作几乎是不可能完成的任务。

企业端的情况也不乐观。HR部门在做招聘计划时，需要知道市场上同类岗位的薪资中位数、竞争对手的开价范围、哪些技能是当下最热门的。这些信息分散在多个平台上，逐一浏览整理既耗时又容易遗漏。更棘手的是，招聘市场的变化速度很快——上周还紧缺的岗位，这周可能就已经饱和了。如果依赖周报或月报这种低频统计，决策永远滞后于市场。

求职者面临的信息不对称问题同样突出。一个计算机专业的应届生想了解"Java开发工程师"在全国各城市的薪资分布，或者在考虑要不要学Python时想看看市场上Python相关岗位的增长趋势，目前并没有什么趁手的工具能直接给出答案。多数人只能凭感觉和零散的帖子做判断，结果就是薪资预期要么太高要么太低，投递的岗位要么超出能力范围要么屈才了。

上述痛点的本质是同一个问题：招聘数据虽然量大、维度多，但缺乏一个能把数据从各个平台汇聚起来、清洗成统一格式、再用直观的图表展示出来的工具，更不用说在此基础上叠加AI智能分析的能力了。这就是我做这个系统的出发点。

### 1.2 国内外研究现状

#### 1.2.1 国外现状

海外招聘信息化比国内早了大约十年。LinkedIn从2003年开始运营，到今天已经积累了超过9亿的会员档案和数千万的活跃职位。Glassdoor则走出了另一条路——让员工匿名填写薪资和公司评价，从而构建了一个薪资透明化的数据库。Indeed作为全球最大的职位聚合搜索引擎，每天处理来自数万个雇主网站的职位信息。

在技术层面，国外的研究走得更远一些。LinkedIn的求职推荐系统基于协同过滤和图神经网络，能同时考虑候选人的技能图谱和人脉关系。Glassdoor的薪资估算模型则融合了用户自报数据和企业公开数据，通过贝叶斯平滑来消除样本偏差。Indeed的岗位搜索引擎底层用到了Elasticsearch的向量检索能力，支持语义级别的岗位匹配——即使用户输入的表述和职位描述不完全一致，也能匹配上。

不过这些系统有一个共同的局限：它们是围绕欧美市场的招聘规则设计的。美国的岗位类型（比如Software Engineer和Data Scientist的分类方式）、薪资结构（年薪制为主）、以及工作签证等特殊维度，放到国内招聘场景里并不适用。更关键的是，这些平台都是高度商业化的SaaS产品，其核心技术不对外公开，也无法低成本地移植到国内环境来使用。

#### 1.2.2 国内现状

国内的招聘平台在数据规模上有天然优势——中国每年有超过1000万的应届毕业生需要找工作，加上跳槽、转行等需求，活跃求职者群体保守估计超过1亿人。BOSS直聘、智联招聘、前程无忧三大平台各自积累了数亿的注册用户和千万级的活跃职位。

但从数据利用的深度来看，国内平台目前主要还是停留在"信息展示+简单搜索"的阶段。BOSS直聘的"看准"板块提供了一些薪资参考数据，但覆盖面和更新频率都有限；智联招聘的"薪智"报告通常是季度发布的静态PDF，时效性跟不上市场变化。大多数平台提供的筛选条件虽然不少，但组合起来做交叉分析的能力仍然不足。

在学术研究和开源项目方面，近年来也出现了一些招聘数据分析的尝试。部分研究者使用Python的Scrapy框架抓取特定平台的职位数据，再用Pandas做统计分析，最后用Matplotlib出图。这种方案在技术上是可行的，但通常只能覆盖单个平台、特定岗位类型，缺少一个能把多平台数据整合到一起并提供交互式分析界面的一体化系统。另外，将大语言模型能力融入招聘数据分析场景的工作还处于非常早期的阶段。

### 1.3 选题意义

#### 1.3.1 理论意义

做这个项目的过程中，我需要把好几项原本独立运作的技术串联到一起：WebMagic负责调度爬虫任务、Jsoup负责解析HTML、正则表达式负责数据清洗、Spring Boot负责提供API、Vue+ECharts负责前端展示、智谱AI负责自然语言问答。每一项技术单独拿出来都有大量的教程和文档，但要把它们有机地整合成一个能实际运行的系统，中间有不少工程层面的问题需要解决——比如爬虫抓取到的数据怎么和数据库的schema对齐、AI模型的SSE流式输出怎么透传到浏览器端、权限校验怎么做才能既安全又不过度复杂。这个项目从需求分析到架构设计再到编码实现的全过程，为"如何在一个招聘场景下把数据采集、数据治理、可视化分析和AI服务这四件事做闭环"提供了一个可以复用的参考案例。

#### 1.3.2 实践意义

从实用角度来看，这个系统至少能帮到三类人：

**第一类是企业HR和管理者。** 他们可以通过系统的可视化分析界面，快速看到某个岗位在不同城市的薪资中位数、不同学历层次的分布比例、最热门的技能标签排行。这些信息能直接辅助他们做薪资定价和岗位描述的撰写，避免开价脱离市场行情。

**第二类是求职者。** 尤其是即将毕业的大学生——他们往往对"市场到底需要什么技能""我的专业能拿到多少薪水"这些问题缺乏清晰认知。系统提供的薪资预测和岗位推荐功能，能帮他们把模糊的求职目标转化成具体的数字和方向。

**第三类是做人才市场研究的学者或分析师。** 他们需要结构化的、可查询的招聘数据来做趋势分析。系统每天凌晨2点自动运行定时爬取任务，持续积累数据，能为这类研究提供数据底座。

### 1.4 设计目标与实现方法

#### 1.4.1 设计目标

我在开始编码之前，给自己定了几个具体的、可量化的目标：

1. **数据源覆盖广。** 至少支持6个以上的招聘平台，每个平台都能通过配置关键词和城市来灵活采集。
2. **数据质量可控。** 入库数据的重复率控制在0.5%以下，薪资、学历等核心字段的标准化准确率不低于98%。
3. **可视化分析维度多。** 至少支持城市分布、薪资区间、技能排行、企业热度、学历要求、经验要求这6个分析维度，所有图表能跟随筛选条件联动刷新。
4. **AI能力可用。** 智能问答要支持流式输出（打字机效果），岗位推荐和薪资预测要在5秒内返回结果。
5. **系统响应快。** 页面加载不超过3秒，查询接口不超过1秒，图表渲染不超过2秒。
6. **并发稳定。** 50个用户同时访问时不出现超时或服务不可用。

#### 1.4.2 实现方法

围绕上述目标，我选择了以下技术路径：

- **架构层面**采用前后端分离。后端是一个纯粹的JSON API服务器，前端是一个SPA单页应用，两者通过HTTP协议交互。这样做的好处是前后端可以独立开发、独立部署，后端改了接口逻辑不会影响前端的构建产物。
- **后端选型**上，Java 22 + Spring Boot 3.2.5是我最熟悉的技术栈。MyBatis 3.0.4的XML映射方式让我对SQL有完全的控制权，这在做复杂的统计查询时很重要。WebMagic框架虽然不算最新的爬虫工具，但它的模块化设计（Downloader、PageProcessor、Scheduler、Pipeline四个组件各司其职）很适合这种多平台并行爬取的场景。
- **前端选型**上，Vue 3的Composition API写起来比Options API更灵活，TypeScript能在编译期就 catch 到很多类型错误。Element Plus提供了开箱即用的表格、表单、弹窗等后台管理常用组件，省去了大量样式工作。ECharts在图表类型和配置灵活性上远超其他同类库。
- **数据采集**的具体做法是：为每个招聘平台写一个独立的Jsoup解析器（实现统一的JobParser接口），解析器内部针对该平台的HTML结构做DOM选择。各平台的CSS选择器、字段映射规则都封装在各自的Parser类中，新增一个平台只需要新增一个Parser实现类。
- **AI服务**的实现没有使用智谱AI官方的Java SDK，而是直接用HttpURLConnection发起HTTP POST请求。原因是SDK引入了大量我不需要的依赖，而且和Spring Boot的SseEmitter配合时遇到了一些兼容性问题。用原生HTTP调用反而更可控。

### 1.5 论文结构

全文一共8章。第1章交代选题的来龙去脉和整体目标；第2章逐项介绍用到的技术和工具，重点说明选型理由而不是简单罗列名词；第3章从技术和业务两个角度论证项目能不能做、值不值得做；第4章把需求拆解成功能需求、数据需求和性能需求三个维度来描述；第5章画出系统的整体架构和数据库表结构；第6章是最核心的一章，按模块逐个展开详细设计和关键代码的说明；第7章给出测试用例和测试结果；第8章做总结，同时坦诚地列出系统的不足和后续可以改进的方向。

**Postman**用来做API调试的"裁判"。在开发每个Controller接口时，我会先在Postman里构造请求、确认返回的JSON结构和状态码都正确，然后再去写前端对应的调用代码。**Navicat Premium**在数据库设计和SQL调试阶段用得很多，特别是写那些涉及多表JOIN和GROUP BY的统计SQL时，能在图形界面里直接看到执行结果，比在日志里翻JSON快得多。**Git**负责版本管理，我采用了"功能分支"策略——每做一个新模块就开一个分支，做完合并回main。后端用**Maven**管理依赖和打包，前端用**Vite**做开发和构建，Vite的热更新速度比Webpack快了一个数量级，改一行CSS就能在浏览器里立即看到效果。

### 2.2 后端开发技术

#### 2.2.1 Java语言

选Java 22而不是更早的LTS版本（比如Java 17），主要是看重了它在Record Pattern、String Templates等特性上的改进。虽然这个项目里用到的Java 22新特性不多，但Virtual Threads（虚拟线程）在爬虫场景下很有想象空间——如果每个爬取请求都分配一个虚拟线程而不是平台线程，并发量可以大幅提升。不过当前版本还是用了传统的FixedThreadPool，Virtual Threads留给后续优化。

#### 2.2.2 Spring Boot框架

Spring Boot 3.2.5在这个项目里承担了三件事：第一，自动配置——只要在pom.xml里引入`spring-boot-starter-web`、`spring-boot-starter-security`、`mybatis-spring-boot-starter`这几个starter依赖，框架就会自动完成组件扫描、数据源配置、安全过滤链搭建等繁琐工作；第二，内嵌Tomcat——打包成一个jar文件就能直接`java -jar`启动，不需要单独部署应用服务器；第三，依赖注入——通过`@Autowired`或构造器注入把Mapper、Service等组件串起来，降低了模块之间的耦合。

#### 2.2.3 MyBatis框架

我选择MyBatis而不是Spring Data JPA，核心原因是：这个项目里有不少复杂的统计SQL——比如技能标签字段的拆分统计、薪资区间的分段聚合、带多个可选筛选条件的动态查询——这些SQL用手写比用JPA的方法名推导或@Query注解要清晰得多。MyBatis的XML映射文件让我可以在一个地方集中管理所有SQL，搭配`<if>`、`<where>`、`<foreach>`等动态标签，能很方便地根据前端传来的筛选参数拼出不同的查询语句。

#### 2.2.4 WebMagic爬虫框架

WebMagic把一个爬虫拆成了四个组件：**Downloader**负责下载页面，**PageProcessor**负责解析页面并提取数据，**Scheduler**负责管理待爬URL队列，**Pipeline**负责处理提取到的数据（在我的系统里就是写入数据库）。这种模块化的设计意味着我可以在不改动其他组件的情况下替换掉任何一个——比如将来想换成Selenium来处理JavaScript渲染的页面，只需要换一个Downloader实现。在实际使用中，我并没有完全照搬WebMagic的标准流程，而是以它为基础，在CrawlServiceImpl中手动控制了线程池和任务调度逻辑。

#### 2.2.5 Jsoup

Jsoup做的事情很纯粹：给定一个HTML字符串（或者URL），用CSS选择器语法来选取DOM节点，然后提取文本或属性值。比如BOSS直聘的职位卡片可能对应一个`<div class="job-card-box">`节点，岗位名称在`.job-title`子节点里，薪资在`.salary`子节点里——用Jsoup写就是`document.select(".job-card-box")`、`card.select(".job-title").text()`、`card.select(".salary").text()`这样的链式调用。每个平台的页面结构不同，所以我为每个平台都写了一个独立的Parser类。

#### 2.2.6 其他工具类

**Lombok**用在实体类上，一个`@Data`注解就能自动生成getter、setter、toString、equals和hashCode，省了大量样板代码。**EasyExcel 3.3.4**处理Excel的导入导出——管理员可以上传一个包含岗位数据的xlsx文件，后端解析后批量入库；也可以把查询结果导出为Excel下载。**SpringDoc OpenAPI**自动从Controller代码中生成API文档，省去了手写接口文档的麻烦。**Spring Validation**通过`@NotBlank`、`@Min`、`@Max`等注解在请求进入Controller之前就完成参数校验，避免无效数据传播到Service层。

### 2.3 前端开发技术

#### 2.3.1 Vue 3

Vue 3最吸引我的特性是Composition API。在2.x版本里，一个组件的逻辑被强制分散到data、methods、computed、watch等选项中，当一个组件变得复杂之后，要来回跳着看代码。Composition API允许我把相关的逻辑组织到一个`setup()`函数或者一个单独的composable文件里，比如可以把所有和ECharts图表相关的逻辑抽成一个`useChart.js`。此外，Vue 3的响应式系统基于Proxy而不是Object.defineProperty，能检测到对象属性的新增和删除，这在动态更新图表数据时避免了之前版本的一些边界bug。

#### 2.3.2 Element Plus

Element Plus 2.7.0提供了我在后台管理界面中需要的几乎所有UI元素：`el-table`用来展示岗位列表和日志列表、`el-form`配合`el-input`/`el-select`做筛选条件表单、`el-dialog`做弹窗编辑、`el-card`做首页的统计卡片布局、`el-menu`做左侧导航栏。它的主题定制也很方便——通过CSS变量就能调整全局配色，我在App.vue里定义了一套CSS变量体系来实现主题切换。

#### 2.3.3 Vue Router

Vue Router 4.3.0在项目中承担两个职责：第一是页面路由——定义好每个路径对应的组件，用户在地址栏输入`/analysis`就跳到可视化分析页面，输入`/ai-chat`就跳到AI服务页面；第二是路由守卫——在`beforeEach`钩子里检查用户是否已登录、是否有权限访问目标页面，没有的话重定向到登录页。

#### 2.3.4 Pinia

Pinia 2.1.7在本项目里存储了两类数据：一类是用户相关的全局状态（登录状态、用户名、角色信息），另一类是分析页面的全局筛选条件（用户在可视化页面选了某个城市和某个学历之后，切换到推荐页面时这些条件应该保留）。相比Vuex，Pinia的API更简洁，不需要写mutations，直接在action里修改state就行。

#### 2.3.5 Axios

Axios 1.7.0在项目中做了一层封装：请求拦截器自动把JWT Token塞到请求头的`Authorization`字段里；响应拦截器统一处理了401（未登录→跳转登录页）、403（无权限→提示）、500（服务端异常→弹错误消息）这几种常见的错误码。这样前端业务代码里就不需要到处写`try-catch`和错误处理逻辑了。

#### 2.3.6 ECharts

ECharts 5.5.0是整个系统"可视化"能力的核心。我最终用到了6种图表类型：饼图展示城市分布、柱状图展示薪资区间、横向柱状图展示技能排行榜、柱状图展示企业招聘热度、双柱状图对比学历和经验分布、折线图展示时间趋势。所有图表都配置了tooltip（鼠标悬停显示详细数据）、legend（图例点击切换系列）、dataZoom（拖拽缩放时间范围）等交互功能。图表的配色方案我选了一组偏商务风的蓝绿色调，通过渐变色填充让图表看起来更有层次感。

### 2.4 数据存储技术

#### 2.4.1 MySQL数据库

选MySQL 8.0.37没太多犹豫——它是目前最成熟稳定的关系型数据库之一，对中文的utf8mb4字符集支持完善，窗口函数和CTE（公共表表达式）在8.0版本中都已可用，社区资源丰富。数据库名取为`recruitment_db`，一共5张核心业务表，外加Spring Security自动生成的几张表。大部分表的InnoDB引擎、utf8mb4字符集、`utf8mb4_general_ci`排序规则在建表DDL中显式指定。

#### 2.4.2 数据库连接池

HikariCP是Spring Boot默认的连接池，号称"最快的JDBC连接池"。在`application.yml`中的关键配置是：`maximum-pool-size: 20`（最大连接数）、`minimum-idle: 5`（最小空闲连接数）、`connection-timeout: 30000`（获取连接的超时时间30秒）、`idle-timeout: 600000`（空闲连接10分钟后回收）。这些参数是根据预期的并发量（50用户同时在线）来设定的——20个连接足以应对这个量级的请求，而不会过多占用数据库资源。

### 2.5 数据采集与清洗技术

#### 2.5.1 网络数据采集

数据采集的流程是这样的：管理员在前端页面选择目标平台、输入关键词和城市、点击"开始采集"，前端POST到`/crawl/task/start`接口，后端在CrawlServiceImpl中创建一条crawl_task记录（状态设为PENDING），然后交给FixedThreadPool（2个线程）去执行。每个线程根据平台类型找到对应的JobParser实现类，用Jsoup下载并解析页面，把提取到的Job对象交给Pipeline写入数据库。

为了应对各平台的反爬措施，我做了四件事：

1. **User-Agent轮换。** 预定义了10条常用的浏览器UA字符串（Chrome、Firefox、Edge各占几条），每次请求随机选一条。
2. **随机延时。** 两次请求之间的间隔在12秒到20秒之间随机取值，模拟人工浏览行为。
3. **失败重试。** 如果某次请求失败（超时或返回非200状态码），最多重试4次，每次重试的等待时间按指数递增（1秒、2秒、4秒、8秒）。
4. **请求头模拟。** 设置Referer字段为目标平台的首页URL，部分平台还设置了Cookie。

系统总共覆盖了8个招聘平台：BOSS直聘、智联招聘、前程无忧、猎聘、拉勾网、牛客网、应届生求职网、领英中国。每个平台的CSS选择器和字段映射规则各不相同，都封装在各自的Parser类中。

#### 2.5.2 数据清洗技术

原始数据到手之后，需要经过一系列清洗才能入库。主要的清洗逻辑如下：

**薪资解析。** 不同平台对薪资的写法差异很大。我写了一个正则表达式 `(\d+(?:\.\d+)?)\s*([万千kK])?\s*[-~到]\s*(\d+(?:\.\d+)?)\s*([万千kK])?` 来统一处理。这个正则能匹配"15K-25K""10000-20000元""1万-2万""15-25K/月"等各种写法。对于匹配结果中的单位字符，K和k乘以1000，万乘以10000，没有单位的默认为"元"。解析后的数值存入job表的min_salary和max_salary两个字段，单位统一为"元/月"。

**学历标准化。** 把所有形如"本科及以上""统招本科""大学本科""Bachelor及以上"的表述统一映射为"本科"，"硕士及以上""研究生""Master"统一映射为"硕士"，以此类推。这是通过一个静态的Map映射表来实现的。

**经验标准化。** "1-3年""一年到三年""1~3yr"统一为"1-3年"，"经验不限""应届生"映射为"应届生"。

**去重。** 对每个岗位的标题+公司名+城市+薪资+平台这几个字段拼接后做SHA-256哈希，再将哈希值做Base64URL编码，得到一个43字符的字符串作为job_key。入库时如果job_key已存在（UNIQUE约束），就跳过该条记录。

**空值过滤。** 如果岗位名称、公司名称、薪资这三个核心字段为空或null，直接丢弃该条记录。

### 2.6 数据可视化技术

可视化分析模块在ECharts之上做了两层封装。第一层是后端的统计接口——每种分析维度对应一个`@GetMapping`端点，比如`/job/stat/city`返回各城市的岗位数量、`/job/stat/salary`返回各薪资区间的岗位数量。这些接口内部通过MyBatis的XML映射执行GROUP BY聚合SQL。第二层是前端的图表组件——每个图表对应一个Vue组件，在`onMounted`时调用后端接口获取数据，然后调用`echarts.init()`初始化图表实例并`setOption()`渲染。

特别值得一提的是技能统计的SQL实现。job表的skills字段存的是逗号分隔的技能标签（比如"Java,Spring Boot,MySQL"），要统计每种技能出现的频次，需要先把这个字段拆开。我在SQL中用了一个SUBSTRING_INDEX配合内联数字表的技巧：

```sql
SELECT skill, COUNT(*) as count FROM (
    SELECT 
        SUBSTRING_INDEX(SUBSTRING_INDEX(j.skills, ',', n.num), ',', -1) AS skill
    FROM job j
    JOIN (SELECT 1 as num UNION SELECT 2 UNION SELECT 3 UNION SELECT 4 UNION SELECT 5) n
    WHERE n.num <= 1 + LENGTH(j.skills) - LENGTH(REPLACE(j.skills, ',', ''))
) t
WHERE skill != ''
GROUP BY skill
ORDER BY count DESC
LIMIT 20
```

这条SQL不需要在Java代码中做任何遍历，完全在数据库层面完成了拆分和聚合，效率比逐行解析高很多。

### 2.7 AI智能服务技术

#### 2.7.1 AI智能问答

智谱AI的GLM-4模型通过HTTPS API调用。我没有使用官方提供的Java SDK，而是直接用`java.net.HttpURLConnection`发起POST请求。原因是：SDK引入的传递依赖太多了（包括OkHttp、Gson等），和项目已有的依赖有版本冲突风险；而且SDK的流式输出接口和Spring Boot的SseEmitter配合时需要额外的适配代码，用原生HTTP反而更简单。

调用流程是这样的：后端收到前端发来的聊天消息后，先查询7种统计数据（城市分布、薪资分布、技能排行、学历分布、经验分布、企业排行、岗位总数）作为上下文，然后把这些统计数据拼接到system prompt里发送给GLM-4。这样AI在回答用户关于招聘数据的问题时，就有了真实的统计数据作为依据，而不是靠"编造"来回答。

流式输出的实现：后端创建一个`SseEmitter`（超时设为120000毫秒即2分钟），在一个异步线程中通过HttpURLConnection读取GLM-4的响应流，每读到一段SSE事件数据（`data: {...}`格式），就通过emitter发送给前端。前端使用fetch API的`response.body.getReader()`逐块读取，把接收到的文本片段拼接到对话内容中，然后用marked.js渲染成Markdown。

#### 2.7.2 岗位智能推荐

推荐算法的核心公式是：

```
matchScore = skillMatch × 0.7 + eduMatch × 0.2 + expMatch × 0.1
```

其中skillMatch是用户输入的技能标签集合和岗位的skills字段之间的Jaccard相似度——即两个集合的交集大小除以并集大小。eduMatch和expMatch则通过预定义的匹配规则来计算：如果用户学历高于等于岗位要求则得1.0，低一级得0.5，低两级得0.0；经验匹配的逻辑类似。所有岗位按matchScore降序排列，取前10条返回。

#### 2.7.3 薪资预测

薪资预测的逻辑分两步。第一步，从数据库中查出与用户输入条件（城市+学历+经验+技能）最接近的岗位群体，计算这些岗位的薪资中位数作为基准值。第二步，在此基础上叠加调整因子：城市因素权重40%（一线城市系数1.2、新一线1.0、二线0.8）、学历因素权重25%（硕士系数1.15、本科1.0、大专0.85）、经验因素权重35%（3年以上1.2、1-3年1.0、应届生0.8）、技能溢价（如果用户填写的技能属于当前热门技能，系数+0.05到+0.15）。最终返回的是一个薪资区间（min和max），以及每个因子的调整明细。

## 3 可行性研究

### 3.1 系统逻辑模型

如果把整个系统画成一个从左到右的数据流图，大概是这样的：

**输入端：** 管理员在爬虫管理页面配置采集任务（选平台、填关键词和城市），或者用户在AI问答页面输入自然语言问题。

**处理端：** 爬虫模块下载页面→Jsoup解析→正则清洗→去重→写入MySQL；统计模块读取MySQL→GROUP BY聚合→组装成ECharts需要的JSON格式；AI模块查数据库取统计数据→拼system prompt→调GLM-4 API→SSE流式返回。

**输出端：** 可视化页面的ECharts图表、AI问答页面的打字机效果文本、推荐页面的岗位列表及匹配度分数、预测页面的薪资区间。

五层结构自下而上分别是：

1. **数据采集层**——负责到外部平台去"搬砖"，把非结构化的HTML转化成半结构化的Java对象。
2. **数据处理层**——做"质检"，清洗、标准化、去重，把半结构化的对象变成可以入库的干净数据。
3. **数据存储层**——MySQL数据库，所有干净数据都存在这里，是后续查询和分析的唯一数据源。
4. **业务服务层**——Spring Boot的Service层，根据前端的需求去数据库里查数据、做计算、调AI，然后把结果包装成统一的Result对象返回。
5. **前端展示层**——Vue 3把后端返回的数据渲染成表格、图表、对话气泡等可视化形式。

### 3.2 可行性分析

#### 3.2.1 技术可行性

我在开始这个项目之前，已经具备了Java、Spring Boot、MyBatis、Vue、MySQL这些技术的基本开发能力。WebMagic和Jsoup虽然之前没用过，但它们的API设计很直观，官方文档也写得清楚，花了一周时间熟悉之后就上手了。智谱AI的GLM-4 API遵循标准的HTTPS+JSON协议，有详细的接口文档和错误码说明，接入难度不高。ECharts虽然配置项很多，但官方的示例页面覆盖了绝大多数常见图表类型，照着改就能用。总体来说，所用到的每一项技术都是成熟的、有充分社区支持的，技术上不存在跨越不了的障碍。

#### 3.2.2 操作可行性

系统是一个Web应用，用户只需要打开浏览器输入地址就能访问。普通用户的使用路径是：登录→看图表→问AI问题→看推荐结果，全程不需要任何技术背景。管理员多了一个"创建爬虫任务"的操作，也是在一个下拉框里选平台、在输入框里填关键词和城市、点一下按钮，三步就完成了。界面布局参考了常见的后台管理系统风格（左侧导航+右侧内容），学习成本很低。

#### 3.2.3 经济可行性

开发过程中用到的所有工具和框架都是开源免费的：Java和MySQL有GPL/商业双许可但免费使用，Spring Boot是Apache 2.0许可，Vue和ECharts是MIT许可。IDE用的是Cursor（免费版就够用），数据库管理用Navicat Premium有教育版。部署方面，一台4核8G的云服务器月费大约在100元以内，学生优惠还能更便宜。相比系统能带来的价值（帮助了解市场行情、优化招聘决策），这个成本是可以忽略不计的。

#### 3.2.4 法律可行性

系统采集的数据来自各招聘平台的公开招聘页面——也就是说，这些信息是平台已经主动发布到互联网上的、任何人在浏览器里输入URL就能看到的内容。采集过程中控制了请求频率（12-20秒的随机间隔），不会对目标平台的服务器造成过大压力。所有采集到的数据仅用于个人的毕业设计学习和数据分析研究，不用于任何商业目的。在爬虫管理页面的免责声明中也明确标注了数据来源和使用限制。

## 4 系统需求分析

### 4.1 功能需求

系统有两个角色：普通用户（USER）和管理员（ADMIN）。两者的权限有明确的区分。

#### 4.1.1 普通用户功能

普通用户登录系统之后，可以执行以下操作：

**（1）查看可视化分析。** 这是普通用户最常使用的功能。进入分析页面后，页面上方有一组筛选条件（关键词输入框、城市下拉框、学历下拉框、经验下拉框），下方是6个图表区域：城市岗位分布饼图、薪资区间分布柱状图、技能需求排行横向柱状图、企业招聘热度柱状图、学历要求分布图、经验要求分布图。当用户修改了任何一个筛选条件，所有图表都会重新请求数据并刷新。用户还可以把图表导出为PNG图片。

**（2）浏览和搜索岗位列表。** 提供一个带分页的表格，每页显示20条记录。表格的列包括：岗位名称、公司名称、所在城市、薪资范围、学历要求、经验要求、来源平台、发布时间。表格上方有和可视化页面类似的筛选条件，但多了一个"薪资范围"的双端滑块筛选。

**（3）使用AI智能服务。** 包括三个子功能：
- **AI问答：** 一个对话式的界面，用户输入问题（比如"北京Java开发的平均薪资是多少"），AI基于数据库中的真实统计数据来回答，回答过程以打字机效果逐字显示。
- **岗位推荐：** 用户填写自己的技能（多个，用逗号分隔）、学历、工作经验年限、期望城市，系统返回最匹配的10个岗位，每个岗位标注匹配度百分比。
- **薪资预测：** 用户选择城市、学历、经验、输入技能，系统返回一个预测的薪资区间以及各因子的影响说明。

**（4）个人操作。** 包括登录、注册、退出登录、修改密码。

#### 4.1.2 管理员功能

管理员拥有普通用户的所有功能，另外增加了以下管理权限：

**（1）爬虫任务管理。** 在任务管理页面可以创建新的爬取任务——选择目标平台（8个平台可选）、输入关键词和城市，点击提交。页面的下半部分是一个任务列表，显示历史任务的平台、关键词、城市、状态（PENDING/RUNNING/DONE/FAILED）、采集到的岗位数量、创建时间和完成时间。管理员可以删除已完成或失败的任务。

**（2）招聘数据管理。** 提供比普通用户更完整的数据操作能力：查看所有岗位数据、多条件组合筛选、单条或批量删除、Excel格式批量导入（通过EasyExcel解析上传的xlsx文件）、Excel格式导出查询结果。还可以手动触发一轮数据清洗——对全量数据重新执行标准化和去重。

**（3）用户管理。** 查看注册用户列表，包含用户名、角色、邮箱、注册时间、最后登录时间。管理员可以删除用户、切换用户的启用/禁用状态、修改用户角色（在USER和ADMIN之间切换）。

**（4）系统日志查看。** 日志记录了每个用户的操作行为：谁在什么时间、从什么IP地址、访问了哪个接口、操作是否成功。管理员可以按时间范围和操作类型筛选日志，也可以导出日志为Excel文件。

### 4.2 数据需求

系统需要持久化的数据分为五类，分别对应五张数据库表：

**（1）用户数据（user表）。** 每个注册用户一条记录。关键字段包括：用户名（唯一）、密码（SHA-256加盐哈希后存储，原始密码从不落地）、盐值（每个用户独立随机生成）、角色（ADMIN或USER）、邮箱、创建时间。密码字段长度设为VARCHAR(255)是因为Base64编码的SHA-256哈希值加上盐值的拼接结果大约是这个长度。

**（2）企业数据（company表）。** 每个被采集到的企业一条记录。字段包括：企业名称、所属行业、所在城市、企业规模、创建时间。企业数据是在爬虫解析过程中从岗位页面间接提取的——职位卡片上通常包含公司名称，系统据此在company表中查找或创建对应的记录。

**（3）岗位数据（job表）。** 这是整个系统中数据量最大、最核心的一张表。每条记录对应一个被采集到的招聘职位。字段包括：关联的企业ID、岗位名称、来源平台标识、43字符的SHA-256唯一键job_key（UNIQUE约束，用于去重）、工作城市、经验要求、学历要求、最低薪资和最高薪资（DECIMAL(10,2)类型，单位：元/月）、逗号分隔的技能标签、岗位描述文本、原始发布时间、最近一次抓取时间、入库创建时间。

**（4）爬虫任务数据（crawl_task表）。** 每次管理员创建一个采集任务就插入一条记录。字段包括：来源平台、搜索关键词、目标城市、任务状态（PENDING/RUNNING/DONE/FAILED四个枚举值）、实际采集到的岗位数量、描述信息、创建时间和完成时间。

**（5）系统日志数据（sys_log表）。** 每次用户操作触发日志记录时插入一条。字段包括：操作用户名、操作名称（如"创建爬虫任务""查询岗位列表"）、请求的URI路径、客户端IP地址、操作是否成功（0或1）、创建时间。

### 4.3 性能需求

我在设计阶段就明确了以下性能指标，并在开发过程中持续关注：

**（1）响应速度。** 静态资源（HTML、JS、CSS、图片）通过浏览器缓存和Vite的代码分割策略，保证首次加载后二次访问几乎秒开。后端API接口——普通的CRUD查询响应控制在500毫秒以内，涉及GROUP BY的统计查询控制在1秒以内。AI问答的首Token延迟控制在3秒以内（受限于外部API的网络往返时间）。ECharts图表在接收到数据后的渲染时间不超过1秒。

**（2）并发处理。** 系统需要能同时服务50个在线用户。HikariCP连接池的20个最大连接数足以覆盖这个量级——按照"每个请求平均占用连接200毫秒"来估算，20个连接每秒可以处理100个请求，而50个用户在正常使用场景下的QPS通常不会超过20。

**（3）数据容量。** job表的设计目标是能存储100万条以上的岗位记录而不出现明显的查询变慢。为此，在job_key、city、education、source_site等常用筛选字段上都建立了索引。分页查询使用PageHelper的MySQL方言（LIMIT offset, size），在大偏移量下通过条件索引来避免全表扫描。

**（4）稳定性。** 系统应该能够连续运行7天不出现服务中断。爬虫模块在单个任务失败时不影响其他任务的执行——每个任务运行在独立的线程中，异常被try-catch捕获后记录日志，不会传播到主线程。

### 4.4 运行环境需求

#### 4.4.1 硬件环境

服务器端（开发阶段用本机替代）的最低配置要求是：4核CPU、8GB内存、100GB硬盘空间。这个配置可以同时运行Spring Boot应用和MySQL数据库而不会出现资源瓶颈。客户端只需要一台能运行现代浏览器的电脑，没有其他特殊硬件要求。

#### 4.4.2 软件环境

服务器端需要安装：JDK 22（运行Spring Boot应用）、MySQL 8.0（数据存储）、Maven 3.6+（后端项目构建）。可选的Nginx反向代理用于生产环境的部署。客户端只需要Chrome 80+或Edge 80+浏览器，不需要安装任何插件或额外软件。

## 5 系统概要设计

### 5.1 软件结构设计

#### 5.1.1 前端层

前端是一个基于Vue 3的单页应用（SPA），用Vite做构建工具。整个前端工程按功能拆分为以下页面组件：

- **Login.vue** — 登录和注册页面，包含表单校验和JWT Token管理。
- **Dashboard.vue** — 系统首页，展示4个统计卡片（总岗位数、总企业数、今日新增、覆盖平台数）和一个最近采集任务的简要列表。
- **Analysis.vue** — 数据可视化分析页面，包含6个ECharts图表组件和筛选条件表单。

- **AIChat.vue** — AI智能服务页面，包含问答、推荐、预测三个Tab。
- **UserManage.vue** — 用户管理页面，管理员专用。

所有页面共享一个左侧导航栏（App.vue中的el-menu组件）和一个顶部状态栏。路由配置在router/index.ts中，路由守卫在router/guard.ts中。

### 5.2 功能模块设计

系统按职责划分为六个模块：

**（1）认证与权限模块。** Spring Security过滤链拦截所有HTTP请求，JWT Token通过`JwtAuthenticationFilter`验证。未认证的请求访问受保护资源时返回401。已认证的请求通过`@PreAuthorize`注解控制角色权限——比如`@PreAuthorize("hasRole('ADMIN')")`标注的方法只有ADMIN角色才能访问。

**（2）爬虫采集模块。** CrawlServiceImpl管理爬虫的完整生命周期：创建任务→选择Parser→启动线程→执行采集→清洗数据→入库→更新任务状态。这个模块是系统中最复杂的，代码量也最大（1425行）。

**（3）数据管理模块。** JobServiceImpl提供岗位数据的增删改查、分页查询、动态条件筛选、Excel导入导出等功能。所有查询都走MyBatis的动态SQL，根据前端传来的参数决定WHERE子句的内容。

**（4）可视化分析模块。** 后端通过JobMapper中的统计方法执行聚合查询，前端通过ECharts组件渲染结果。筛选条件通过Pinia全局状态管理，确保在可视化页面和推荐页面之间切换时条件不丢失。

**（5）AI智能服务模块。** AIServiceImpl封装了所有AI相关逻辑：问答（SSE流式调用GLM-4）、推荐（权重匹配算法）、预测（统计+因子调整）。所有AI功能在调用前都会先查一次数据库获取最新的统计数据作为上下文。

**（6）系统管理模块。** 包括用户管理（UserServiceImpl）和日志管理（LogServiceImpl）。日志通过Service层手动记录——在关键操作执行成功后，Service代码中显式调用`sysLogMapper.insert()`写入日志。

#### 5.3.3 完整性约束

- **主键约束：** 每张表都有自增的BIGINT主键id。
- **唯一约束：** user表的username字段、job表的job_key字段各设了UNIQUE索引，从数据库层面杜绝重复。
- **外键关联：** job表的company_id指向company表的id。实际实现中没有在DDL中加FOREIGN KEY约束（因为外键约束在数据量大时会影响写入性能），而是在Service层的业务逻辑中保证引用完整性。
- **非空约束：** 每张表的"关键"字段都标注了NOT NULL——user表的username和password、job表的title和job_key、crawl_task表的status等。这些约束确保不会出现"没有名字的用户"或"没有标题的岗位"这种脏数据。
- **索引优化：** 除了主键和唯一索引之外，在job表的city、education、source_site、publish_time字段上也建了普通索引，因为这些字段是统计查询和筛选查询中最常用的过滤条件。

## 6 系统详细设计与实现

### 6.1 数据爬取管理功能

#### 6.1.1 功能描述

这个模块是整个系统的数据入口。管理员通过前端界面配置采集参数（选平台、填关键词和城市），后端接收请求后创建一条crawl_task记录，然后在一个独立的线程池中启动实际的爬取流程。采集完成后更新任务状态和采集数量。此外，系统还配置了一个Spring的@Scheduled定时任务，每天凌晨2点自动对所有平台执行一轮预设关键词的采集。

#### 6.1.2 界面设计

爬虫管理页面上半部分是一个卡片区域，包含：el-select下拉框（列出8个平台选项）、el-input输入框（关键词）、el-input输入框（城市）、el-button提交按钮。下半部分是el-table表格，列定义包括：任务ID、来源平台、关键词、城市、状态（用el-tag标签显示不同颜色——PENDING灰色、RUNNING蓝色、DONE绿色、FAILED红色）、采集数量、创建时间、操作列（删除按钮）。

#### 6.1.3 实现逻辑

当管理员点击提交按钮时，前端发送POST请求到`/api/crawl/task/start`，请求体包含sourceSite、keyword、city三个字段。后端CrawlController接收到请求后：

1. 参数校验——确认sourceSite不为空且在预设的8个平台列表中。
2. 在crawl_task表中插入一条记录，status设为PENDING。
3. 从Spring的ApplicationContext中根据sourceSite获取对应的JobParser Bean。
4. 把解析任务提交到FixedThreadPool（corePoolSize=2）中异步执行。

在线程的run方法内部，执行流程如下：
- 把crawl_task的status更新为RUNNING。
- 调用Parser的parseJobs方法，传入Jsoup解析后的Document对象，得到List<Job>。
- 对每个Job执行清洗操作：解析薪资文本→标准化学历和经验→生成job_key。
- 逐条尝试插入company表（INSERT IGNORE，已存在则跳过）和job表（同样利用job_key的UNIQUE约束去重）。
- 成功插入的计数器累加，失败（重复）的记录跳过。
- 每个平台单次采集上限为200条（SITE_LIMIT常量），达到上限后停止。
- 全部完成后，更新crawl_task的status为DONE，job_count为实际入库数量。

如果采集过程中抛出异常（网络超时、页面结构变化导致解析失败等），catch块会更新任务status为FAILED，并在message字段中记录异常信息。

定时任务的实现是在CrawlServiceImpl上加了一个`@Scheduled(cron = "0 0 2 * * ?")`注解的方法。每天凌晨2点，该方法遍历预设的"关键词×城市"组合，对每个组合在每个平台上创建并执行采集任务。如果某次真实爬取因为网络原因没有拿到任何数据，系统会生成20到35条模拟数据作为备份——这些模拟数据的薪资、学历、经验等字段从实际已入库数据的统计分布中随机生成，确保不会出现"图表全空"的情况。

public class BossParser implements JobParser {

        Elements cards = document.select(".job-card-box");
        int count = 0;

            SalaryRange range = SalaryParseUtil.parseSalary(salaryText);
            job.setMinSalary(range.getMin());
            job.setMaxSalary(range.getMax());
            job.setCity(card.select(".job-area").text());
            job.setEducation(card.select(".tag-list .edu").text());
            job.setExperience(card.select(".tag-list .exp").text());
            job.setSkills(card.select(".tag-list .skill").text());
            job.setSourceSite(getPlatform());
            // 基于核心字段生成去重键
            job.setJobKey(JobKeyUtil.generate(
                getPlatform(), job.getTitle(),
                job.getCompanyName(), job.getCity(),
                job.getMinSalary(), job.getMaxSalary()));
            jobs.add(job);
            count++;
        }
        return jobs;
    }
}

薪资解析正则工具类的核心逻辑：

```java
public class SalaryParseUtil {
    // 匹配 "15K-25K" "10000-20000" "1万-2万" "15~25K/月" 等写法
    private static final Pattern SALARY_PATTERN = Pattern.compile(
        "(\\d+(?:\\.\\d+)?)\\s*([万千kK])?\\s*[-~到]\\s*" +
        "(\\d+(?:\\.\\d+)?)\\s*([万千kK])?"
    );

    public static SalaryRange parseSalary(String text) {
        SalaryRange range = new SalaryRange();
        if (text == null || text.contains("面议")) return range;
        Matcher m = SALARY_PATTERN.matcher(text);
        if (m.find()) {
            range.setMin(convertToYuan(
                Double.parseDouble(m.group(1)), m.group(2)));
            range.setMax(convertToYuan(
                Double.parseDouble(m.group(3)), m.group(4)));
        }
        return range;
    }

    private static double convertToYuan(double value, String unit) {
        if (unit == null) return value;
        switch (unit) {
            case "K": case "k": return value * 1000;
            case "万": return value * 10000;
            default: return value;
        }
    }
}
```

技能统计SQL——用SUBSTRING_INDEX实现逗号分隔字段的拆分统计：

```xml
<select id="statBySkill" resultType="map">
    SELECT skill, COUNT(*) as count FROM (
        SELECT
            TRIM(SUBSTRING_INDEX(
                SUBSTRING_INDEX(j.skills, ',', n.num), ',', -1
            )) AS skill
        FROM job j
        JOIN (
            SELECT 1 as num UNION ALL SELECT 2 UNION ALL
            SELECT 3 UNION ALL SELECT 4 UNION ALL SELECT 5
        ) n ON n.num &lt;= 1 + LENGTH(j.skills) - LENGTH(REPLACE(j.skills, ',', ''))
        <where>
            <if test="keyword != null and keyword != ''">
                AND (j.title LIKE CONCAT('%',#{keyword},'%')
                     OR j.skills LIKE CONCAT('%',#{keyword},'%'))
            </if>
            <if test="city != null and city != ''">
                AND j.city = #{city}
            </if>
        </where>
    ) t
    WHERE skill != ''
    GROUP BY skill
    ORDER BY count DESC
    LIMIT 20
</select>
```

### 6.4 AI智能服务功能

#### 6.4.1 功能描述

AI智能服务是系统中最具"差异化"的模块。它不是简单的"调API然后返回结果"，而是在调用大模型之前先做了一层数据准备——把数据库中最新的7类统计数据查出来，拼接到system prompt里，让AI的回答有真实数据支撑。

该模块包含三个子功能，在AIChat.vue页面中通过el-tabs切换：

- **智能问答（AI Chat）：** 对话式交互，支持Markdown渲染和代码高亮。
- **岗位推荐（Recommend）：** 填表→点按钮→看结果列表的模式。
- **薪资预测（Salary Predict）：** 和推荐类似的填表模式。

#### 6.4.2 界面设计

**AI问答界面**采用经典的聊天布局：左侧是历史对话列表（可切换不同对话），右侧是主聊天区域——上方滚动区域显示消息列表（用户消息靠右、AI消息靠左），下方是输入框和发送按钮。AI消息支持Markdown渲染（通过marked.js）和代码块语法高亮。

**岗位推荐界面**分上下两部分：上面是el-form表单（技能输入框、学历下拉框、经验下拉框、城市输入框、推荐按钮），下面是推荐结果列表——每个结果是一个el-card，展示岗位名称、公司、城市、薪资、匹配度进度条。

**薪资预测界面**的布局和推荐界面类似，但结果区域展示的是一个薪资区间卡片（大字体显示"最低薪资 ~ 最高薪资"），下方列出各因子的调整明细。

#### 6.4.3 实现逻辑

**AI问答的完整调用链路：**

1. 用户在前端输入问题，点击发送。前端把消息POST到`/api/ai/chat/stream`。
2. 后端AIServiceImpl收到请求后，先执行"上下文增强"步骤：连续调用7个统计方法（statTotal、statByCity、statBySalary、statBySkill、statByEducation、statByExperience、statByCompany），拿到当前的招聘市场全貌数据。
3. 把统计数据格式化为文本，拼接到system prompt中。system prompt的大意是："你是一个招聘数据分析助手。以下是当前系统中的真实统计数据：[统计数据文本]。请基于这些数据回答用户的问题。如果数据中没有相关信息，请如实说明。"
4. 构建JSON请求体（model="glm-4"、messages数组包含system和user两条），通过HttpURLConnection向智谱AI的API端点发起POST请求。
5. API返回的是SSE格式流——每行以`data: `开头，后面跟一个JSON对象，包含choices[0].delta.content字段（模型输出的文本片段）。
6. 后端在while循环中逐行读取响应流，每读到一段content，就通过SseEmitter发送给前端。
7. 前端通过fetch API的ReadableStream逐块接收，把文本片段拼接到当前消息的内容变量中，然后通过marked.js渲染为HTML，实现打字机效果。

**岗位推荐的实现：**

1. 用户填写个人信息（技能、学历、经验、城市），前端POST到`/api/ai/recommend`。
2. 后端先从job表中查出与用户条件匹配的候选岗位（city和education作为硬性过滤条件）。
3. 对每个候选岗位计算三项分数：
   - skillScore：把用户输入的技能列表和岗位的skills字段都转为Set，计算Jaccard系数 = |A∩B| / |A∪B|。
   - eduScore：用户学历≥岗位要求得1.0，低一级得0.5，低两级得0.0。
   - expScore：逻辑同eduScore。
4. 最终得分 = skillScore × 0.7 + eduScore × 0.2 + expScore × 0.1。
5. 按得分降序排列，取前10条返回。

**薪资预测的实现：**

1. 用户选择条件，前端POST到`/api/ai/predict`。
2. 后端从job表中查出满足城市+学历+经验条件的岗位群体，计算薪资中位数作为基准值。
3. 叠加调整因子：
   - 城市系数：北京/上海/深圳/广州/杭州 = 1.2，成都/武汉/南京/西安/苏州 = 1.0，其他 = 0.85。
   - 学历系数：硕士 = 1.15，本科 = 1.0，大专 = 0.85。
   - 经验系数：5年以上 = 1.2，3-5年 = 1.1，1-3年 = 1.0，应届生 = 0.8。
   - 技能溢价：如果用户填写的技能在TOP20热门技能中，每命中一个+0.03，上限+0.15。
4. 最终预测区间 = [基准min × 综合系数 × 0.9, 基准max × 综合系数 × 1.1]，上下留10%的浮动空间。

@PostMapping("/stream")
public SseEmitter streamChat(@RequestBody AIChatRequest request) {
    // 120秒超时，覆盖大模型长回答的场景
    SseEmitter emitter = new SseEmitter(120000L);

    executor.execute(() -> {
        try {
            // 1. 查7类统计数据作为上下文
            String context = buildStatContext();
            // 2. 构建请求体
            String requestBody = buildGLMRequest(context, request.getMessage());
            // 3. 发起HTTP请求
            HttpURLConnection conn = (HttpURLConnection)
                new URL("https://open.bigmodel.cn/api/paas/v4/chat/completions")
                .openConnection();
            conn.setRequestMethod("POST");
            conn.setDoOutput(true);
            conn.setRequestProperty("Content-Type", "application/json");
            conn.setRequestProperty("Authorization", "Bearer " + apiKey);

            try (OutputStream os = conn.getOutputStream()) {
                os.write(requestBody.getBytes(StandardCharsets.UTF_8));
            }

            // 4. 逐行读取SSE流
            try (BufferedReader reader = new BufferedReader(
                    new InputStreamReader(conn.getInputStream()))) {
                String line;
                while ((line = reader.readLine()) != null) {
                    if (line.startsWith("data: ") && !line.contains("[DONE]")) {
                        String json = line.substring(6);
                        JSONObject obj = JSON.parseObject(json);
                        String content = obj.getJSONArray("choices")
                            .getJSONObject(0)
                            .getJSONObject("delta")
                            .getString("content");
                        if (content != null) {
                            emitter.send(SseEmitter.event().data(content));
                        }
                    }
                }
            }
            emitter.complete();
        } catch (Exception e) {
            emitter.completeWithError(e);
        }
    });

    return emitter;
}

岗位推荐算法核心：

```java
public List<JobRecommendVO> recommend(String skills, String education,
                                       Integer experience, String city) {
    // 从数据库查出候选岗位
    List<Job> candidates = jobMapper.selectByCondition(city, education, null);
    Set<String> userSkills = new HashSet<>(Arrays.asList(skills.split(",")));

    List<JobRecommendVO> results = new ArrayList<>();
    for (Job job : candidates) {
        Set<String> jobSkills = new HashSet<>(
            Arrays.asList(job.getSkills().split(",")));
        // Jaccard相似度
        Set<String> intersection = new HashSet<>(userSkills);
        intersection.retainAll(jobSkills);
        Set<String> union = new HashSet<>(userSkills);
        union.addAll(jobSkills);
        double skillMatch = union.isEmpty() ? 0 :
            (double) intersection.size() / union.size();

        // 学历匹配
        double eduMatch = calcEduMatch(education, job.getEducation());
        // 经验匹配
        double expMatch = calcExpMatch(experience, job.getExperience());

        double score = skillMatch * 0.7 + eduMatch * 0.2 + expMatch * 0.1;
        results.add(new JobRecommendVO(job, score));
    }

    return results.stream()
        .sorted(Comparator.comparingDouble(
            JobRecommendVO::getScore).reversed())
        .limit(10)
        .collect(Collectors.toList());
}
```

### 6.5 系统管理功能

#### 6.5.1 功能描述

系统管理模块包含用户管理和日志管理两个子功能。这两个功能只有ADMIN角色可以访问，通过`@PreAuthorize("hasRole('ADMIN')")`注解在Controller方法上控制。

#### 6.5.2 界面设计

**用户管理界面：** 一个el-table表格，列包括用户名、角色（el-tag标签显示）、邮箱、注册时间、操作（删除按钮、角色切换按钮）。表格上方有搜索框可以按用户名模糊查询。

**日志管理界面：** 上方是筛选条件——时间范围（el-date-picker类型为datetimerange）、操作类型下拉框、查询按钮、导出按钮。下方是日志表格，列包括日志ID、操作用户、操作名称、请求URI、IP地址、是否成功（成功绿色标签/失败红色标签）、操作时间。日志数据量通常较大，固定每页50条。

#### 6.5.3 实现逻辑

**用户管理：** UserController提供`GET /api/user/list`（分页查询用户列表）、`DELETE /api/user/{id}`（删除用户）、`PUT /api/user/{id}/role`（修改角色）。删除用户时会同步删除该用户的所有日志记录。角色切换只能在ADMIN和USER之间进行。

**日志管理：** 日志的记录方式是在Service层的业务方法中手动调用——当某个关键操作执行完成后，Service代码中构造一个SysLog对象，设置username、action、uri、ip等字段，然后调用`sysLogMapper.insert(sysLog)`写入数据库。这种方式比AOP切面更灵活，因为可以在记录日志时附带操作相关的上下文信息（比如"删除了ID为123的岗位"这样的描述）。

LogController提供`GET /api/log/list`（分页查询+筛选）、`GET /api/log/export`（导出Excel）。日志查询通常需要扫描大量记录，所以在created_at字段上建立了索引来加速按时间范围查询。

### 6.6 数据库访问实现

#### 6.6.1 实现逻辑

MyBatis在本项目中的使用遵循"接口+XML"的模式。每个实体类对应一个Mapper接口和一个XML映射文件。Mapper接口定义方法签名，XML文件中写具体的SQL语句。

MyBatis的配置集中在`application.yml`中：

```yaml
mybatis:
  mapper-locations: classpath:mapper/*.xml
  type-aliases-package: com.example.recruitment.entity
  configuration:
    map-underscore-to-camel-case: true   # 数据库下划线→Java驼峰自动映射
    log-impl: org.apache.ibatis.logging.stdout.StdOutImpl  # 开发环境打印SQL日志
```

`map-underscore-to-camel-case: true`这个配置很关键——它让MyBatis自动把数据库的`min_salary`字段映射到Java对象的`minSalary`属性，不需要在XML中逐个字段写`<result>`映射。

分页查询通过PageHelper实现。在Service方法中调用`PageHelper.startPage(pageNum, pageSize)`之后，紧接着执行的第一个MyBatis查询会被自动拦截并添加`LIMIT`子句。查询结果包装在`PageInfo<T>`对象中，包含total（总记录数）、pages（总页数）、list（当前页数据）等分页信息。

#### 6.6.2 关键代码

统计查询的XML映射（薪资区间统计为例）：

```xml
<select id="statBySalary" resultType="map">
    SELECT
        CASE
            WHEN max_salary &lt; 5000 THEN '5K以下'
            WHEN max_salary &lt; 10000 THEN '5K-10K'
            WHEN max_salary &lt; 15000 THEN '10K-15K'
            WHEN max_salary &lt; 20000 THEN '15K-20K'
            WHEN max_salary &lt; 30000 THEN '20K-30K'
            ELSE '30K以上'
        END AS range_name,
        COUNT(*) AS count
    FROM job
    <where>
        <if test="keyword != null and keyword != ''">
            AND (title LIKE CONCAT('%',#{keyword},'%')
                 OR skills LIKE CONCAT('%',#{keyword},'%'))
        </if>
        <if test="city != null and city != ''">
            AND city = #{city}
        </if>
    </where>
    GROUP BY range_name
    ORDER BY MIN(max_salary)
</select>
```

这段SQL用CASE WHEN把连续的薪资值离散化为6个区间，然后GROUP BY统计每个区间的岗位数量。ORDER BY MIN(max_salary)确保区间按从低到高的顺序排列。

## 7 系统测试

### 7.1 测试目的与内容

测试的目的不是"走过场"，而是要在交付之前尽可能多地发现问题。我给自己设定的测试覆盖范围是：所有公开的API接口（至少每个Controller的方法都测一次）、所有前端页面的核心交互路径、以及几个关键的性能指标。

测试内容分为四类：

**（1）功能正确性。** 验证每个功能是否按照需求规格说明书中的描述正常工作。比如：创建爬虫任务后任务状态是否正确设为PENDING、筛选查询是否真的过滤了数据、AI问答是否能返回合理的内容等。

**（2）数据准确性。** 验证数据处理环节的正确性——薪资解析是否准确（比如"15K-25K"应该解析出min=15000、max=25000）、去重逻辑是否有效（同一个岗位不会重复入库）、统计结果是否和手工计算一致。

**（3）性能指标。** 验证系统在正常负载下的响应速度是否符合设计目标。

**（4）异常处理。** 验证系统在异常输入（空关键词、不存在的城市、超大页码等）和异常场景（数据库连接断开、外部API超时）下是否优雅降级而不是崩溃。

### 7.2 测试方法

根据不同的测试目标，我采用了不同的测试方法：

**功能测试**主要用黑盒方法——把系统当作一个整体，只关注"输入什么、期望输出什么"，不关心内部实现。测试时直接在浏览器中操作系统，手动输入各种参数组合，观察返回结果是否符合预期。这种方法的优点是贴近真实使用场景，缺点是覆盖面受限于测试用例的设计。

**接口测试**用Postman完成——每个API端点都构造了正常请求和异常请求两种case，检查返回的HTTP状态码和JSON结构是否正确。异常请求包括：缺少必填参数、参数类型错误、传入不存在的ID等。

**性能测试**用JMeter模拟并发场景。设置了50个并发线程同时请求`/api/job/page`接口，持续运行5分钟，观察是否有超时、错误或响应时间显著上升的情况。

**兼容性测试**在Chrome 124和Edge 124两个浏览器上进行，同时测试了1920×1080和1366×768两种常见屏幕分辨率下布局是否正常。

## 结论

这个毕业设计做下来，我最大的感受是：把一个系统从零做到能跑起来，中间遇到的工程问题远比想象的多。需求文档里写"支持多平台数据采集"只有8个字，但真正做起来需要为每个平台写独立的HTML解析器、调试CSS选择器、处理各种页面结构变化、设计合理的重试机制、考虑去重策略——这些都是文档里看不到的工作量。

从结果来看，系统基本实现了我在需求分析阶段设定的目标：覆盖了8个招聘平台的数据采集、建立了一套可复用的清洗流程（正则解析+标准化映射+SHA-256去重）、提供了6个维度的交互式可视化分析、集成了AI问答/推荐/预测三项智能服务。在性能方面，实测数据也全部达标。

不过坦诚地说，这个系统还有不少可以改进的地方：

**第一，爬虫的稳定性还需要加强。** 招聘平台的页面结构不是一成不变的——BOSS直聘在开发期间就改过一次职位卡片的CSS类名，导致解析器突然拿不到数据，我不得不重新分析页面结构并更新选择器。更健壮的做法是做"选择器版本管理"——每次解析前先检测页面的DOM结构是否符合预期，不符合时自动切换到备用选择器或触发告警。

**第二，AI服务的可靠性受限于外部API。** 智谱AI的API偶尔会出现响应超时或返回空内容的情况，这种异常不在我的控制范围内。后续可以考虑增加一个本地的小型NLP模型（比如基于BERT的轻量级问答模型）作为降级方案——当外部API不可用时自动切换到本地模型，虽然效果差一些但至少不会完全不可用。

**第三，数据分析的深度还可以拓展。** 目前的技能统计只是简单计数（某个技能出现了多少次），如果能做"技能组合分析"（比如"Java + Spring Boot + MySQL"这个组合出现了多少次、对应什么薪资水平），对求职者的指导意义会更大。又比如加入时间维度，展示某个技能的需求量在过去几个月的变化趋势，用折线图呈现出来。

**第四，权限体系还不够精细。** 目前ADMIN和USER之间是一个粗粒度的二分——管理员要么能看到所有数据，要么什么管理功能都用不了。更合理的做法可能是做细粒度的权限控制（RBAC），比如"数据分析师"角色只能看图表和导出数据、"内容管理员"角色只能管理岗位数据和爬虫任务、"系统管理员"才能管理用户和查看日志。

这些改进方向不是本阶段能全部完成的，但它们构成了清晰的后续迭代路径。总体而言，这个项目验证了"数据采集+数据治理+可视化分析+AI服务"这套技术组合在招聘场景下的可行性，也为我后续的技术成长打下了不错的基础。
