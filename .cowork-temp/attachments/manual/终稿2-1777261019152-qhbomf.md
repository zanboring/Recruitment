# AI招聘数据可视化系统的设计与实现（终稿2）

## 摘要

在数字化转型持续推进的背景下，招聘活动正在从“信息发布驱动”走向“数据决策驱动”。企业在招聘过程中需要同时面对多平台数据分散、字段口径不一致、人工分析效率低、策略复盘困难等现实问题；求职者则普遍存在岗位筛选成本高、薪资预期缺乏依据、技能提升方向不明确等痛点。针对上述问题，本文以“数据采集-数据治理-可视化分析-AI辅助服务”为主线，设计并实现了一个面向招聘场景的 AI 招聘数据可视化系统。

系统采用前后端分离架构。后端基于 Java 22、Spring Boot 3.2.5、MyBatis 3.0.4 构建，结合 Spring Security、JWT、PageHelper、Jsoup、EasyExcel 等技术实现认证授权、任务化爬取、数据治理、统计聚合和导入导出功能；前端基于 Vue 3、TypeScript、Element Plus、Pinia、Vue Router、Axios、ECharts 实现管理与分析页面；数据存储采用 MySQL 8.0，使用 HikariCP 连接池保障并发访问稳定性。系统同时接入智谱 GLM-4 接口，提供同步问答与 SSE 流式问答能力，并基于本地实时统计上下文增强回答相关性。

在功能层面，系统实现了用户认证、岗位查询、爬虫任务管理、数据导入导出、多维度统计分析、岗位推荐、薪资预测、AI分析报告等核心能力。测试与运行结果表明：系统在教学规模数据下具有较好的功能完整性、交互可用性与可维护性，能够为企业招聘策略优化和求职者职业决策提供数据支持。本文工作验证了“规则化数据治理+可视化分析+大模型增强”在招聘场景中的可行性，为后续扩展为生产级系统提供了工程基础。

**关键词**：招聘数据；数据可视化；前后端分离；Spring Boot；Vue 3；数据爬取；AI问答；岗位推荐；薪资预测

## Abstract

With the rapid growth of digital transformation, recruitment activities are shifting from information publishing to data-driven decision-making. Enterprises face practical challenges such as fragmented multi-platform data, inconsistent field standards, low manual analysis efficiency, and weak strategy review capabilities. Job seekers also encounter difficulties in position filtering, salary expectation calibration, and skill planning. To address these problems, this thesis designs and implements an AI-powered recruitment data visualization system based on a full workflow of data collection, data governance, visual analytics, and AI-assisted services.

The system adopts a front-end/back-end separated architecture. The back-end is built with Java 22, Spring Boot 3.2.5, and MyBatis 3.0.4, integrating Spring Security, JWT, PageHelper, Jsoup, and EasyExcel for authentication, task-based crawling, data processing, statistical aggregation, and import/export capabilities. The front-end is implemented with Vue 3, TypeScript, Element Plus, Pinia, Vue Router, Axios, and ECharts. MySQL 8.0 is used for persistent storage with HikariCP as the connection pool. The system also integrates GLM-4 APIs to support both synchronous and SSE streaming chat, with context enhancement based on real-time local statistics.

Functionally, the system provides user authentication, job query, crawl task management, data import/export, multi-dimensional visualization, job recommendation, salary prediction, and AI analysis reporting. Experimental results show that the system achieves good functional completeness, usability, and maintainability under graduation-project-level data scale. It can effectively support enterprise recruitment optimization and job seekers' career decision-making. This work verifies the practicality of combining rule-based data governance, visual analytics, and LLM enhancement in recruitment scenarios.

**Key words**: Recruitment Data; Data Visualization; Front-end/Back-end Separation; Spring Boot; Vue 3; Data Crawling; AI Chat; Job Recommendation; Salary Prediction

## 第1章 绪论

### 1.1 研究背景

招聘业务天然具有“高频更新、跨平台分散、强时效”的特征。岗位发布与下线、薪资调整、技能需求变化在短时间内都会显著影响用人决策。传统招聘流程在企业侧通常表现为“平台搜索+人工汇总+经验判断”，在求职者侧表现为“多平台反复检索+主观比较”。该模式在数据量较小时尚可运行，但在岗位规模增长和更新频率提升后，出现以下突出问题：

1. 数据来源分散，平台字段口径不一致，跨平台横向比较困难；
2. 人工统计工作量大，分析结果滞后，难以支撑即时决策；
3. 岗位推荐与薪资判断缺乏量化依据，结论解释性不足；
4. 数据难以沉淀为结构化资产，无法形成可复用的分析闭环。

在 AI 技术快速发展的背景下，招聘系统已经不再仅仅承担“信息展示”功能，而应进一步具备“数据治理、趋势解释、决策辅助”的综合能力。因此，设计一个可运行、可维护、可扩展的招聘数据智能分析系统，具有明确的现实意义。

### 1.2 国内外研究现状

国外招聘平台在数据资产化与算法应用方面起步较早，典型平台如 LinkedIn、Indeed 在推荐、画像、趋势分析等能力上具备长期积累。其优势主要体现在规模化用户行为数据、成熟的模型工程体系和持续迭代的产品机制。

国内招聘平台在用户规模和业务覆盖上同样成熟，但从工程实践与教学落地视角看，仍存在可进一步研究的空间：

1. 跨平台数据融合成本高，字段标准化工作量大；
2. 可视化分析多数停留在静态统计，交互式分析深度不足；
3. AI能力常见于问答入口，缺少与本地业务数据的深度耦合；
4. 部分项目重功能展示、轻稳定性治理，缺乏任务可追踪和异常可恢复机制。

因此，本文不追求抽象层面的模型创新，而强调“工程闭环创新”：将爬取、治理、分析、可视化与 AI 服务统一落地，并通过认证授权、日志审计、状态机、降级策略保障系统可运行性。

### 1.3 研究意义

#### 1.3.1 理论意义

1. 将招聘业务问题抽象为可复用的数据处理链路：采集、清洗、建模、分析、辅助决策；
2. 将前后端分离架构、关系型数据治理、可视化分析、大模型服务进行一体化工程实践；
3. 在教学项目约束下，给出“可解释、可落地、可扩展”的智能化实现路径。

#### 1.3.2 实践意义

1. 为企业招聘提供城市热度、薪资结构、技能需求等量化支撑；
2. 为求职者提供更具依据的岗位推荐与薪资参考；
3. 为教学和科研提供完整案例，覆盖需求分析、架构设计、编码实现、测试验证全流程；
4. 为后续扩展生产级系统提供技术与工程基础。

### 1.4 研究目标与内容

围绕系统建设目标，本文的研究内容包括：

1. 需求建模与角色划分；
2. 前后端分离总体架构设计；
3. 多平台任务化爬取与反爬稳定性策略；
4. 数据标准化、去重与岗位生命周期治理；
5. 多维统计接口与可视化页面联动；
6. AI问答、岗位推荐、薪资预测功能实现；
7. 系统测试、风险分析与迭代方向设计。

## 第2章 相关技术与开发环境

### 2.1 开发环境与工具

本项目主要开发环境如下：

1. 操作系统：Windows；
2. 后端开发：JDK 22、Maven；
3. 前端开发：Node.js、Vite；
4. 数据库：MySQL 8.0；
5. 调试工具：Postman、数据库客户端；
6. 代码管理：Git。

### 2.2 后端技术栈

#### 2.2.1 Spring Boot

后端采用 Spring Boot 3.2.5 搭建 RESTful 服务，利用自动配置机制降低样板代码，重点关注业务逻辑实现。项目采用分层结构：Controller 负责请求接入与参数校验，Service 负责业务编排，Mapper 负责数据访问。

#### 2.2.2 MyBatis + PageHelper

数据访问层采用 MyBatis 3.0.4，实现动态 SQL 与统计聚合查询。分页场景通过 PageHelper 统一处理，减少重复分页逻辑。

#### 2.2.3 Spring Security + JWT

系统采用无状态认证方案。用户登录后获取 JWT，后续请求通过 `Authorization: Bearer <token>` 完成身份校验。通过过滤器解析 token 并注入认证上下文，实现受限接口访问控制。

#### 2.2.4 Jsoup 与任务化爬取

爬取模块核心使用 Jsoup 完成页面请求与 DOM 解析，围绕爬虫任务对象组织执行流程，支持状态流转、重试、随机延时、备用数据兜底等机制，提升演示与运行稳定性。

#### 2.2.5 EasyExcel

系统提供 Excel 数据导入导出能力，采用 EasyExcel 实现结构映射与文件流读写，满足管理端批量数据处理需求。

### 2.3 前端技术栈

#### 2.3.1 Vue 3 + TypeScript

前端采用 Vue 3 组合式 API 与 TypeScript，提升复杂页面开发可维护性与类型安全性。

#### 2.3.2 Element Plus

Element Plus 提供表单、表格、弹窗、分页等通用组件，支撑后台管理页面快速搭建。

#### 2.3.3 Vue Router + Pinia

Vue Router 管理页面路由和登录拦截；Pinia 管理用户状态与聊天状态，保证跨页面数据一致性。

#### 2.3.4 Axios + ECharts

Axios 用于统一请求拦截、鉴权注入和错误处理；ECharts 用于城市、薪资、技能、学历经验等维度图表展示。

### 2.4 数据与AI能力

1. MySQL 8.0：关系型数据存储，支持索引优化和事务控制；
2. GLM-4 接口：实现 AI 问答与流式交互；
3. SSE：提升长文本回答的交互体验；
4. 规则模型：实现岗位推荐和薪资预测的可解释计算。

## 第4章 系统总体设计

### 4.1 架构设计

系统采用前后端分离架构，后端采用典型分层模式：

1. 展示层（Vue）：页面交互和图表渲染；
2. 控制层（Controller）：接口接入与参数接收；
3. 业务层（Service）：业务规则与流程编排；
4. 持久层（Mapper + XML）：SQL 映射与数据库操作；
5. 数据层（MySQL）：结构化存储与索引优化。

### 4.2 模块设计

系统按业务划分六大模块：

1. 用户认证与权限模块；
2. 爬虫任务与数据采集模块；
3. 岗位数据管理模块；
4. 统计分析与可视化模块；
5. AI 智能服务模块；
6. 日志与系统治理模块。

### 4.3 核心业务流程

#### 4.3.1 爬虫任务流程

管理员创建任务后，状态置为 `PENDING`；启动后进入 `RUNNING`；采集完成根据执行结果进入 `FINISHED` 或 `FAILED`。任务执行中记录新增、更新、下架、验证等统计信息。

#### 4.3.2 数据分析流程

用户发起筛选请求后，后端执行动态 SQL 聚合，前端将结果映射为图表，形成“查询-统计-展示”闭环。

#### 4.3.3 AI 服务流程

用户提问后，后端先汇总实时统计上下文，再调用 AI 接口。流式场景下通过 SSE 增量推送，前端实时渲染。

### 4.4 数据库设计

#### 4.4.1 核心数据表

1. `user`：用户账户与角色；
2. `company`：企业基础信息；
3. `job`：岗位核心信息；
4. `crawl_task`：爬虫任务管理；
5. `sys_log`：系统日志。

#### 4.4.2 关键字段

1. `job_key`：岗位唯一键，支撑去重；
2. `job_status`：岗位状态（`NEW/ACTIVE/OFFLINE`）；
3. `last_seen_at`：最近一次抓取时间；
4. `min_salary/max_salary`：数值化薪资字段，便于统计与预测。

#### 4.4.3 索引策略

对岗位表的城市、发布时间、状态、来源站点等高频过滤字段建立索引，对 `job_key` 建唯一索引，提升查询和去重效率。

## 第5章 系统详细设计与实现

### 5.1 认证与权限模块

系统提供登录、注册、修改密码和用户信息读取接口。登录成功后下发 token，前端持久化并由请求拦截器自动注入。路由守卫在进入受限页面前校验登录态，后端对权限不足请求返回 403。

密码存储采用“哈希+盐值”方案，避免明文存储风险。后端统一处理未认证和未授权响应，降低前端异常分支复杂度。

### 5.2 爬虫任务模块

#### 5.2.1 任务机制

任务对象记录来源平台、关键词、城市、状态、采集数量、执行消息和时间信息。通过任务机制将长耗时采集操作从 HTTP 请求线程中解耦，提升接口响应稳定性。

#### 5.2.2 稳定性策略

1. 随机 User-Agent；
2. 请求超时控制；
3. 最多多次重试；
4. 随机间隔请求；
5. 失败时备用数据兜底；
6. 关键日志记录。

#### 5.2.3 数据解析与过滤

系统按平台配置解析逻辑并提供通用兜底，提取岗位名称、公司、薪资、城市、学历、经验、技能、发布时间等字段。为提升数据可用性，写库前会过滤无效岗位、异常薪资和不符合定位的数据。

#### 5.2.4 去重与生命周期治理

系统通过 `job_key` 实现幂等写入：不存在则新增，存在则更新。对于本轮未再次出现的岗位进行下线标记，实现岗位生命周期维护。

### 5.3 岗位数据管理模块

岗位查询支持关键词、城市、学历、经验、来源、薪资区间、状态等组合筛选，查询结果支持分页返回。管理员可执行新增、编辑、删除与批量管理操作。

该模块结合动态 SQL 和分页插件，实现了较高灵活性和较低重复代码量。

### 5.4 数据导入导出模块

#### 5.4.1 导入

系统支持 Excel 导入，解析后按字段映射岗位对象并入库。导入流程对关键字段进行校验，防止无效数据污染主数据集。

#### 5.4.2 导出

系统支持按条件导出岗位数据为 Excel 文件，满足数据留档与离线分析需求。

#### 5.4.3 数据清理

提供管理员可用的数据清理接口，用于清空岗位和任务数据。该操作属于高风险操作，需权限控制并建议配合备份流程。

### 5.5 统计分析与可视化模块

#### 5.5.1 统计接口

后端提供城市统计、技能统计、薪资区间统计、学历统计、经验统计、状态统计、热门岗位统计等接口。统计口径尽量统一，避免跨页面同名指标结果不一致。

#### 5.5.2 前端图表实现

前端以 Dashboard 与 Analysis 页面为主要承载，使用 ECharts 渲染多图联动结果。页面支持筛选交互、空态提示、加载状态反馈，保证可读性和可用性。

### 5.6 AI 智能服务模块

#### 5.6.1 AI 问答

系统支持同步问答和流式问答。流式问答场景通过 SSE 增量推送文本，显著改善用户等待体验。

#### 5.6.2 上下文增强

后端在调用模型前注入数据库统计信息（城市、技能、薪资、学历、经验、趋势、热门岗位），使回答更贴近系统当前数据。

#### 5.6.3 岗位推荐

推荐逻辑基于技能匹配分值排序，并结合城市等条件过滤返回候选岗位。该方法实现成本低、可解释性强，适合项目阶段目标。

#### 5.6.4 薪资预测

预测采用“数据库样本均值 + 多因子修正”策略，因子主要来自城市、经验、学历和高薪技能。该方式具备可解释、计算快和易维护特点。

### 5.7 系统治理：日志与异常处理

系统采用统一异常处理机制，降低页面层错误处理复杂度。关键操作会写入日志，便于排障与审计。前端请求层针对 401、403、500、超时等场景给出差异化提示，提升用户理解成本。

## 第6章 系统测试与结果分析

### 6.1 测试目标与环境

测试目标是验证系统功能正确性、交互可用性和基础稳定性。测试环境包括 Windows、JDK 22、MySQL 8.0、Node.js/Vite、主流浏览器。

### 6.2 功能测试

#### 6.2.1 认证与权限

## 第8章 工程实现深度复盘

### 8.1 从功能闭环到工程闭环

在项目早期，实现“能看见页面、能返回接口、能画出图表”并不困难。真正的难点在于系统能否在异常场景、连续运行场景和多模块联动场景下保持可用。本项目从中后期开始将重点从“功能堆叠”转向“工程闭环”，核心体现为：

1. 统一返回结构与统一错误处理；
2. 统一认证与权限校验链路；
3. 任务状态机与可追踪执行日志；
4. 数据治理口径统一；
5. AI 调用前的数据上下文增强。

工程闭环的价值在于可维护、可复盘、可扩展。对于毕业设计而言，这种“可解释的工程完整性”通常比单点功能更有说服力。

### 8.2 控制层与服务层边界复盘

项目重构过程中，对 Controller 与 Service 的职责边界做了明确约束。Controller 层只负责参数接收、基础校验和调用转发，不承载复杂业务逻辑。Service 层负责流程编排、规则处理与异常控制。这样做有三方面收益：

1. 控制器代码更简洁，接口职责更清晰；
2. 服务层逻辑可复用，便于单元测试；
3. 需求变更时改动范围更可控。

例如，岗位筛选逻辑统一下沉至 Service + Mapper 层，前端页面可以在不改业务核心逻辑的前提下快速新增筛选项或调整显示方式。

### 8.3 任务状态机设计合理性分析

若不采用任务状态机，爬虫执行很容易出现“用户触发后页面无反馈、执行失败不可追踪、重复执行冲突”等问题。系统采用 `PENDING -> RUNNING -> FINISHED/FAILED` 的有限状态机设计，配合任务消息字段记录执行摘要，解决了以下问题：

1. 用户可见性：任务状态实时可见；
2. 运维可追踪性：失败原因可定位；
3. 流程一致性：避免重复触发混乱；
4. 演示稳定性：即使失败也有明确反馈。

该设计也是后续升级分布式任务调度的基础接口约定。

### 8.4 数据治理策略复盘

数据治理是本项目质量上限的关键。若数据口径不统一，后续图表和 AI 输出都会失真。项目中主要治理策略如下：

1. **薪资治理**：将文本薪资统一为数值区间，无法识别值不强行估算；
2. **学历治理**：归并同义表达，减少类别碎片化；
3. **经验治理**：统一经验表达口径；
4. **实体治理**：通过 `job_key` 去重，避免重复样本污染；
5. **时效治理**：通过 `job_status` 与 `last_seen_at` 维护岗位生命周期。

治理后，统计与 AI 分析的可信度显著提升，这也是系统从“爬虫演示”走向“分析系统”的本质变化。

### 8.5 统计口径一致性复盘

统计系统常见问题是“同名指标不同口径”。项目通过后端统一聚合接口尽可能保证口径一致，前端多个页面复用同源统计接口。具体做法包括：

1. 对在岗分析默认排除已下线岗位；
2. 热门榜单与分布图共用基础数据源；
3. 页面层只做展示映射，不重复定义统计逻辑。

这种设计减少了“页面A与页面B结果不一致”的风险，也降低了后续维护成本。

### 8.6 AI 增强而非 AI 代替

本项目 AI 模块定位为“分析入口增强”，而非“替代统计引擎”。模型回答前必须注入本地数据库上下文，确保回答与当前数据集相关。该策略相比直接通用问答有明显优势：

1. 回答更贴近本系统数据；
2. 建议更具可执行性；
3. 幻觉风险相对降低；
4. 便于答辩中解释“AI 的依据是什么”。

同时系统保留规则化分析兜底，避免外部 AI 服务异常时系统完全不可用。

### 8.7 前端可维护性复盘

前端在后期重点优化可维护性，核心策略包括：

1. 页面拆分为业务组件，降低单文件复杂度；
2. 请求层统一拦截，减少页面重复代码；
3. 登录态与聊天态统一放入 Pinia；
4. 空态、错态、加载态统一交互体验；
5. 图表初始化与更新逻辑模块化。

这些优化使页面从“能跑”提升到“可持续迭代”。

### 8.8 本章小结

本章复盘说明：项目价值不仅在于功能数量，更在于系统化设计。通过边界清晰的架构、任务状态机、数据治理、统一口径和 AI 上下文增强，系统达到了毕业设计应有的工程深度。

### 9.1 环境准备流程

### 9.2 配置管理实践

项目采用配置外置方式管理关键参数，包括：

1. 数据库账号密码；
2. JWT 密钥与有效期；
3. AI API Key；
4. 爬虫开关与调度开关；
5. 日志级别与文件路径。

配置外置化带来三项收益：环境切换简单、敏感信息不硬编码、故障定位更高效。

### 9.3 典型故障与处理策略

#### 9.3.1 后端启动失败

可能原因：数据库不可用、配置错误、端口冲突、JDK 版本不匹配。  
处理建议：先查日志首个错误栈，再逐项排查依赖项。

#### 9.3.2 登录后接口仍401

可能原因：token 未注入、token 过期、本地状态损坏。  
处理建议：清理本地缓存重新登录，检查请求拦截器。

#### 9.3.3 爬虫任务失败率高

可能原因：外网波动、站点结构变化、访问限制。  
处理建议：观察任务消息和日志，必要时启用备用数据并更新解析规则。

#### 9.3.4 AI 返回异常

可能原因：未配置 API Key、网络不可达、外部服务波动。  
处理建议：检查配置项，先用同步接口验证可达性，再看流式链路。

### 9.4 备份与恢复建议

即便是教学项目，也应建立最基本备份机制：

1. 重要演示前做一次数据库快照；
2. 执行清理操作前再次确认并备份；
3. 保留一套稳定演示数据集；
4. 存档关键截图与日志片段用于论文证据链。

部署与运维内容体现项目“可交付能力”。可复现、可排障、可恢复是工程型毕业设计的重要评价维度。

## 第12章 讨论与展望（扩展）

### 12.1 关于算法取舍的讨论

毕业设计阶段采用规则模型而非复杂深度学习模型，核心原因是：

1. 可解释性强，便于论文论证和答辩陈述；
2. 实现成本可控，符合项目周期；
3. 数据量与标注条件尚不足以支撑高质量训练；
4. 可在现有架构上平滑升级。

该取舍并非降低标准，而是强调“阶段目标与工程现实匹配”。

### 12.2 关于数据质量边界的讨论

系统仅对公开岗位信息进行分析，结论受采集范围、采集时点、平台规则变化影响。论文结论应明确边界，避免夸大结论泛化能力。

### 12.3 关于AI可信性的讨论

AI回答质量与上下文质量高度相关。项目中采用“统计先行、AI解释随后”策略，减少无依据输出。后续可引入引用标注机制，进一步增强回答可核验性。

### 12.4 关于系统演进路径的讨论

推荐三阶段演进：

1. 工程稳态阶段：索引优化、缓存、告警、自动化测试；
2. 能力增强阶段：技能关联表、语义匹配、预测模型升级；
