# AI招聘数据可视化系统的设计与实现（二稿，2万字扩展版）

## 摘要

随着数字经济深入发展，企业人才竞争持续加剧，招聘活动由传统“信息发布驱动”逐步转向“数据与智能决策驱动”。在真实招聘场景中，岗位信息分散在多个平台，字段命名和结构不一致，数据质量参差不齐，导致企业在岗位画像、薪酬定价、人岗匹配、招聘节奏规划等方面仍大量依赖人工经验，难以形成可追踪、可验证、可复用的决策闭环。与此同时，求职者同样面临岗位信息噪声高、职业路径不清晰、薪酬预期缺乏依据等问题。为此，本文围绕“多源招聘数据治理+可视化分析+AI智能服务”展开研究，设计并实现了一个面向实际应用的 AI 招聘数据可视化系统。

系统采用前后端分离架构，后端以 Java 22、Spring Boot 3.2.5、MyBatis 3.0.4 为核心，结合 PageHelper、Spring Security、JWT、EasyExcel、Jsoup 等组件，构建了数据采集、数据治理、数据服务与权限控制能力；前端以 Vue 3、TypeScript、Element Plus、Pinia、Vue Router、ECharts 为核心，完成管理端与分析端的交互实现。系统在数据层基于 MySQL 8.0 完成结构化建模，围绕用户、企业、岗位、爬取任务、系统日志等对象建立关系型存储体系。针对招聘数据“异构、冗余、脏污”的特点，系统实现了岗位去重、薪资归一化、技能标签抽取、学历经验标准化等策略；针对智能分析需求，系统集成 GLM-4 大模型接口，提供 AI 问答、流式对话、岗位推荐、薪资预测、智能分析报告等功能。

在实现层面，本文不仅完成了基础 CRUD 功能，而且将“业务可运行性”作为核心目标：通过任务状态机、重试与降级、抽样验证、日志可追踪、权限隔离等机制提升系统稳定性；通过统计接口与图表联动实现多维分析；通过 AI 上下文增强实现“基于实时数据库状态的回答”。测试结果表明，本系统在功能完整性、响应性能、稳定性和可扩展性方面满足毕业设计场景要求，可为企业招聘策略优化、求职者职业决策和教学科研实践提供可行参考。

**关键词**：招聘数据；数据可视化；前后端分离；Spring Boot；Vue 3；爬虫；人工智能；岗位推荐；薪资预测

## Abstract

With the rapid growth of the digital economy, talent competition among enterprises has become increasingly intense. Recruitment is gradually shifting from an information-publishing model to a data- and intelligence-driven decision model. In real-world scenarios, job postings are distributed across multiple platforms with heterogeneous schemas and inconsistent field standards, resulting in high data governance costs and weak decision support capabilities. To address these challenges, this thesis proposes and implements an AI-powered recruitment data visualization system that integrates multi-source data collection, data cleaning, visual analytics, and intelligent services.

The system adopts a front-end/back-end separated architecture. The back-end is built with Java 22, Spring Boot 3.2.5, and MyBatis 3.0.4, integrating PageHelper, Spring Security, JWT, EasyExcel, and Jsoup for service orchestration. The front-end is developed using Vue 3, TypeScript, Element Plus, Pinia, Vue Router, and ECharts to deliver an interactive and user-friendly interface. MySQL 8.0 is used as the core relational storage, with data models for users, companies, jobs, crawling tasks, and system logs.

A complete data pipeline is implemented, including crawling, normalization, de-duplication, structured storage, statistical aggregation, and chart-based presentation. AI capabilities are integrated via GLM-4 APIs to support intelligent Q&A, SSE-based streaming chat, job recommendation, salary prediction, and AI-enhanced analysis reports. Experimental and functional tests demonstrate that the system is practical, maintainable, and extensible for graduation-project-level deployment and iterative enhancement.

**Key Words**: Recruitment Data; Data Visualization; Spring Boot; Vue 3; Crawler; AI Service; Job Recommendation; Salary Prediction

## 第1章 绪论

### 1.1 研究背景

招聘市场是典型的高频变化场景。岗位新增、岗位下线、薪酬调整、技能需求迁移都具有明显的时间敏感性。传统招聘管理流程大多以“平台搜索+人工筛选+经验判断”为主：企业 HR 每天在多个站点检索岗位和候选人，手工统计投递转化率、岗位竞争度、地区薪资水平；求职者则通过碎片化信息比较职位吸引力。该模式在小规模时可勉强运行，但在岗位规模和更新频率提升后，问题迅速放大：第一，数据来源分散，平台口径不一致，难以统一比较；第二，人工统计效率低，无法形成实时分析；第三，岗位推荐和薪资判断缺乏客观依据；第四，数据结果无法沉淀为组织知识。

在这种背景下，招聘系统的升级方向逐渐明确：一是多源数据的自动采集与标准化治理；二是面向决策场景的可视化分析；三是以 AI 为入口的低门槛交互分析。换言之，系统不应停留在“展示岗位列表”，而应具备“发现规律—解释规律—辅助决策”的能力。

本课题的价值在于将以上三类能力统一到同一系统中，并在工程实现中解决“可运行、可维护、可扩展”问题，而不是停留于概念验证。本文基于真实项目代码，给出从需求分析、架构设计到模块实现、测试验证的完整路径，以期为同类人力资源数字化系统提供可复用方案。

### 1.2 研究意义

#### 1.2.1 理论意义

（1）将招聘业务问题抽象为标准数据处理链路：采集、清洗、建模、分析、决策；  
（2）将前后端分离架构、关系型数据治理、可视化分析与大模型服务进行一体化实践；  
（3）在“教育项目可落地”的约束下，平衡算法复杂度与工程可维护性，形成轻量智能化方案。

#### 1.2.2 实践意义

（1）企业可通过城市分布、岗位热度、薪资结构、技能需求趋势等结果优化招聘策略；  
（2）求职者可通过推荐与预测功能获得更可解释的职业参考；  
（3）教学与科研可基于该系统开展数据治理、可视化工程、AI 应用开发等训练；  
（4）系统具备继续扩展为“人岗匹配中台”或“行业分析平台”的基础。

### 1.3 国内外研究现状

国外招聘平台在数据资产化和智能推荐方面发展较早。以 LinkedIn、Indeed 等为代表，平台不仅支持职位检索，还广泛使用行为特征、语义特征、关系图谱进行个性化推荐和人才画像。其优势在于长期沉淀的用户行为数据和算法基础设施。

国内招聘平台在业务规模、用户活跃度方面已经非常成熟，职位检索、消息沟通、在线测评等能力丰富，但从毕业设计可实现角度看，仍存在可研究空间：  
（1）跨平台数据整合困难，采集策略与字段治理成本高；  
（2）可视化分析常局限在基础统计，难以形成“结论+建议”；  
（3）AI 能力往往以问答为主，缺少与业务数据深度耦合；  
（4）系统工程层面对容错、审计、权限、任务可观测性的关注不足。

本文的创新点不在于提出全新算法，而在于构建“工程可运行”的完整闭环：多平台爬取与任务管理、规则化数据治理、统计分析接口、可视化前端、AI 增强服务、权限与日志治理统一落地。

### 1.4 研究内容与目标

围绕系统目标，本文的研究内容包含以下七个方面：

1. **需求建模**：明确普通用户与管理员角色边界，定义核心用例；  
2. **系统架构设计**：构建前后端分离和分层服务模型；  
3. **数据采集策略设计**：实现多平台采集、重试、降级与反爬策略；  
4. **数据治理设计**：实现薪资、学历、经验、技能的标准化处理与去重；  
5. **可视化分析设计**：构建多维统计接口并映射到图表展示；  
6. **AI 服务设计**：实现流式问答、岗位推荐、薪资预测与分析报告；  
7. **测试与评估**：验证系统功能性、稳定性、可用性与可维护性。

## 第2章 相关技术与开发环境

### 2.1 总体技术选型原则

本系统的技术选型遵循三项原则：  
（1）**成熟稳定优先**：选择主流、社区活跃、文档完善的框架；  
（2）**开发效率优先**：在保证可维护性前提下提高迭代速度；  
（3）**与需求匹配优先**：避免过度设计，重点服务于数据治理与分析流程。

### 2.2 后端技术体系

#### 2.2.1 Java 22 与 Spring Boot 3.2.5

后端采用 Java 22，配合 Spring Boot 快速构建 RESTful 服务。Spring Boot 的自动配置机制显著减少基础样板代码，便于将开发重点放在业务逻辑本身。系统将控制层、服务层、数据访问层分离，实现职责清晰、便于测试与迭代。

#### 2.2.2 MyBatis 3.0.4 与 PageHelper 2.1.0

在数据访问层，系统采用 MyBatis 以获得对 SQL 的精细控制。由于招聘分析依赖大量聚合查询、条件组合查询，手写 SQL 更利于优化可读性和性能。分页场景使用 PageHelper 统一处理，降低分页重复代码。

#### 2.2.3 Spring Security + JWT

系统采用无状态认证模式：用户登录成功后获得 JWT，后续请求通过 `Authorization: Bearer <token>` 完成认证。`JwtAuthFilter` 在过滤器链中解析 Token，注入认证上下文；`SecurityConfig` 统一配置白名单接口、跨域策略、鉴权失败处理。该方式兼顾安全性与前后端分离场景下的会话管理便利性。

#### 2.2.4 Jsoup 爬取解析与任务机制

系统通过 Jsoup 进行页面连接与 DOM 解析，围绕“任务”组织爬取流程。任务包含来源站点、关键词、城市、状态、结果数量、执行消息等。采集执行中加入超时、重试、随机延时、UA 轮换、备用数据降级等机制，以提高稳定性并避免采集失败导致流程中断。

#### 2.2.5 EasyExcel 数据导入导出

管理端需支持数据批量导入导出。EasyExcel 在大文件场景下相比传统方案更节省内存，且对象映射简单，适合课程项目的工程落地。

### 2.3 前端技术体系

#### 2.3.1 Vue 3 + TypeScript

前端采用 Vue 3 组合式 API 与 TypeScript，提升复杂页面状态管理的可维护性。对于岗位列表、分析看板、AI 对话等高交互页面，组合式 API 有利于逻辑拆分与复用。

#### 2.3.2 Element Plus

Element Plus 提供表单、表格、对话框、分页、消息提示等后台常用组件，帮助快速构建一致性较高的管理界面，降低重复 UI 开发成本。

#### 2.3.3 Pinia 与 Vue Router

Pinia 用于管理用户状态、聊天状态等全局数据。Vue Router 负责页面路由与鉴权守卫，未登录用户自动跳转登录页，保证受限页面安全访问。

#### 2.3.4 Axios 与统一拦截器

系统通过 `http.ts` 封装请求与响应拦截：请求阶段自动附加 Token；响应阶段统一解析后端格式；错误阶段按状态码分层处理（401/403/500/超时等），提高调用一致性和可维护性。

#### 2.3.5 ECharts 可视化引擎

ECharts 支持柱状图、饼图、横向条形图、多系列组合图等形式，能够覆盖招聘场景下“分布类、排名类、趋势类、结构类”统计需求。系统在 `Dashboard` 与 `Analysis` 页面实现多图联动展示。

### 2.4 数据与AI技术

#### 2.4.1 MySQL 8.0

数据库使用 MySQL 8.0，字符集采用 `utf8mb4`，支持中文文本与多语言字符。数据表设计兼顾规范化与查询效率：通过主键、唯一键、索引优化岗位检索和统计性能。

#### 2.4.2 大模型服务（GLM-4）与 SSE

AI 服务由后端 `AIService` 封装，支持同步调用与流式调用。流式对话通过 SSE（Server-Sent Events）逐段推送文本，有效改善长回答等待体验。系统在 AI 问答前注入数据库统计上下文，实现“带数据依据”的回答，而非纯通用问答。

## 第3章 系统可行性分析

### 3.1 技术可行性

从技术栈看，本系统全部基于成熟开源组件。Spring Boot、MyBatis、Vue 3、ECharts 均有丰富社区实践。系统关键技术风险主要在“采集稳定性”和“数据治理准确性”，本文通过以下机制降低风险：

- 任务级重试和随机延时减少采集失败；
- 站点选择器多套匹配策略提高解析兼容；
- 启用备用数据降级保证流程可用；
- 字段标准化与去重保证数据质量；
- 抽样验证与状态标记避免历史岗位误判。

综上，技术实现路径清晰、可执行性高。

### 3.2 经济可行性

系统以开源框架为主，开发与部署成本低。对毕业设计或中小团队场景而言，仅需常规开发机和数据库环境即可运行。若面向更大规模应用，可通过分层部署、缓存优化、异步任务队列等逐步扩容，不需要一次性高投入。

### 3.3 操作可行性

系统面向两类用户：  
- **普通用户**：使用岗位检索、可视化分析、AI 功能；  
- **管理员**：管理爬虫任务、数据导入导出、数据清洗与监控。

页面交互采用常见后台范式，学习成本低。系统为关键流程提供操作反馈和错误提示，具备可操作性。

### 3.4 合规可行性

系统采集对象为公开岗位信息，不采集敏感个人信息；同时采用访问频率控制、重试间隔、抽样验证等策略，避免对目标站点造成异常压力。系统输出主要用于教学与分析，不涉及未经授权的数据商业化售卖，具备基本合规边界意识。

## 第4章 需求分析

### 4.1 角色分析

系统角色划分如下：

1. **普通用户**：关注数据查询、图表分析和 AI 辅助；  
2. **管理员**：负责任务创建执行、数据治理和系统维护；

#### 4.2.1 用户认证与权限管理

- 用户注册、登录、修改密码；
- 登录后获取 JWT；
- 受限接口根据角色和权限控制访问；
- 未认证访问返回 401，权限不足返回 403。

#### 4.2.2 数据采集管理

- 创建爬虫任务（来源、关键词、城市）；
- 启动任务并异步执行；
- 记录任务状态、结果数量、执行消息；
- 查询任务列表、删除任务；
- 支持定时任务自动触发。

#### 4.2.3 招聘数据管理

- 岗位分页查询与多条件筛选；
- 岗位新增、更新、删除；
- 数据导入导出（Excel）；
- 数据清洗与异常处理。

#### 4.2.4 可视化分析

- 城市分布、薪资区间、技能热度；
- 学历/经验要求分布；
- 企业招聘排行、热门岗位排行；
- 分析摘要与图表联动。

#### 4.2.5 AI 智能服务

- AI 同步问答；
- AI 流式问答（SSE）；
- 岗位推荐；
- 薪资预测；
- AI 分析报告生成。

### 4.3 非功能需求

#### 4.3.1 性能需求

- 常规分页查询在秒级内返回；
- 图表数据可接受范围内快速渲染；
- AI 流式对话具备可感知的持续输出。

#### 4.3.2 稳定性需求

- 爬虫任务失败后可重试或降级；
- 关键操作有日志与状态可追踪；
- 系统出现异常时不影响核心页面可访问。

#### 4.3.3 安全性需求

- 认证鉴权闭环；
- 密码哈希与盐值存储；
- 接口参数校验与异常拦截；
- 日志避免泄露敏感信息。

#### 4.3.4 可维护性需求

- 模块边界清晰；
- 配置可外部化；
- 代码结构支持后续新增站点和分析维度。

## 第5章 系统总体设计

### 5.1 架构设计

系统采用前后端分离与后端分层架构。

#### 5.1.1 前端层

负责页面展示与交互，核心页面包括：`Login`、`Dashboard`、`JobList`、`DataManage`、`Analysis`、`Settings`、`AIChat`。前端通过统一 API 模块调用后端服务，并将响应数据渲染为图表、表格和对话视图。

#### 5.1.2 接口层（Controller）

后端控制器包括：

- `AuthController`：认证与用户信息；
- `JobController`：岗位 CRUD、统计、推荐、预测、AI分析；
- `CrawlController`：任务创建、启动、查询、删除；
- `DataController`：导入、导出、清洗；
- `AIController`：AI同步与流式问答。

#### 5.1.3 业务层（Service）

业务层负责核心逻辑：

- `CrawlServiceImpl`：多平台采集、解析、去重、下架标记；
- `JobServiceImpl`：统计聚合、推荐计算、薪资预测、分析摘要；
- `DataServiceImpl`：Excel 导入导出、数据清理；
- `UserServiceImpl`：登录注册、密码修改、用户信息；
- `AIService`：调用外部大模型并处理 SSE。

#### 5.1.4 数据层（Mapper + MySQL）

Mapper XML 承载动态 SQL 与聚合 SQL。数据库主表 `job`、`crawl_task`、`user`、`company`、`sys_log` 形成系统核心数据底座。

### 5.2 业务流程设计

#### 5.2.1 数据采集流程

管理员创建任务后，系统进入 `PENDING`；启动后转 `RUNNING`。后台线程按“站点—关键词—城市”三层循环执行采集；每条记录按 `job_key` 去重，已有记录更新状态和字段，新记录插入。任务完成后写入 `FINISHED` 或 `FAILED`，并记录统计消息。

#### 5.2.2 数据分析流程

用户发起筛选后，后端根据条件动态拼接 SQL 返回分页数据；统计接口返回维度聚合结果；前端将数据映射到 ECharts 图表并展示。

#### 5.2.3 AI服务流程

用户输入问题后，系统先汇总实时统计数据（城市、技能、薪资、趋势、热门岗位），将其拼接为上下文，再调用 AI。流式模式下后端逐段推送，前端实时拼接并渲染 Markdown。

### 5.3 数据库设计

#### 5.3.1 核心实体关系

- `user`：系统账户；
- `company`：企业实体；
- `job`：岗位实体（关联企业）；
- `crawl_task`：采集任务；
- `sys_log`：日志审计。

#### 5.3.2 关键字段设计说明

- `job.job_key`：岗位唯一识别键，避免重复写入；
- `job.job_status`：`NEW/ACTIVE/OFFLINE`，支持状态分析；
- `job.min_salary/max_salary`：数值化存储，支持聚合统计；
- `job.skills`：逗号分隔标签，配合 SQL 拆分统计；
- `crawl_task.status`：任务状态机，支撑可观测执行。

#### 5.3.3 索引与约束

在 `job` 表建立城市、发布时间、状态、来源站点等索引；对 `job_key` 建立唯一索引；通过外键和非空约束保证数据一致性。

## 第6章 系统详细设计与实现

### 6.1 后端核心实现

### 6.1.1 用户认证模块实现

`AuthController` 提供登录、注册、修改密码、用户信息接口。登录流程如下：

1. 前端提交用户名和密码；
2. `UserServiceImpl` 查询用户并校验密码哈希；
3. 使用 `JwtUtil` 生成 Token，写入 `UserVO` 返回；
4. 前端保存 Token 到本地与 Pinia 状态。

密码采用 `SHA-256 + Salt` 存储，盐值随机生成。相比明文存储显著降低泄露风险。  
`JwtAuthFilter` 对每次请求解析 Token，并将用户 ID 与权限写入 `SecurityContext`，供后续权限判断和业务日志使用。

### 6.1.2 爬虫任务模块实现

`CrawlServiceImpl` 是系统工程复杂度最高的模块，包含以下关键能力：

#### （1）任务生命周期管理

- 创建任务：状态初始化为 `PENDING`；
- 启动任务：更新为 `RUNNING` 并提交线程池执行；
- 完成任务：写回 `FINISHED/FAILED`、结果数量与消息。

#### （2）多层循环执行模型

执行逻辑按站点、关键词、城市组合遍历，单站点设置上限（`SITE_LIMIT`）控制采集规模，防止任务过载。

#### （3）反爬与容错策略

- 随机 User-Agent；
- 请求超时控制；
- 最多 4 次重试；
- 重试间隔带随机扰动；
- 失败时可启用备用数据生成，保证流程可用。

#### （4）解析与清洗

系统针对 BOSS、智联、51job、猎聘等站点配置多组选择器，同时保留通用解析兜底。解析字段包括标题、公司、薪资、城市、学历、经验、描述、URL、发布时间等。

#### （5）业务过滤与质量控制

- 过滤无效岗位（培训、中介、诈骗等关键词）；
- 过滤高级岗位或不适配岗位；
- 薪资阈值过滤（控制异常高薪）；
- 学历经验标准化；
- `job_key` 去重。

#### （6）状态更新与下架标记

采集到的岗位更新为 `ACTIVE/NEW`，未在本轮出现的岗位按策略标记为 `OFFLINE`。系统还引入“最近岗位抽样验证”机制，减少全量 URL 验证的请求成本。

### 6.1.3 岗位数据服务实现

`JobServiceImpl` 提供查询、统计、推荐、预测、AI反馈聚合等能力。

#### （1）查询与分页

使用 PageHelper 实现分页，`JobMapper.xml` 通过 `<if>` 动态拼装条件，实现关键词、城市、学历、经验、来源、薪资区间、状态等组合筛选。

#### （2）统计分析

通过聚合 SQL 实现：

- 城市维度统计；
- 企业维度统计；
- 技能维度统计（拆分 skills 字段）；
- 薪资区间统计；
- 学历与经验分布；
- 岗位状态统计；
- 热门岗位 TOP10；
- 近7日趋势。

#### （3）薪资预测

预测先取数据库匹配样本平均薪资，再叠加城市、经验、学历、技能溢价因子。  
该方案具备三点优势：可解释、计算快、实现成本低。虽然精度不及复杂机器学习模型，但在课程项目中具有高实用性。

#### （4）岗位推荐

推荐逻辑根据技能匹配程度排序，并结合城市条件筛选。`matchScore` 采用关键词包含匹配，返回前 10 条候选岗位。该方法简单但直观，适合初版系统。

#### （5）AI分析反馈

`analyzeWithAI` 先通过规则引擎汇总优质岗位、技能需求、趋势、薪资分析，再生成结构化 `AIFeedbackVO`。该做法使系统在 AI 不可用时仍能输出结果，提升可靠性。

### 6.1.4 数据导入导出模块实现

`DataServiceImpl` 负责 Excel 导入导出与数据清理：

- 导入：解析 `JobExcelRow`，逐行转 `Job` 并入库；
- 导出：查询限定条目并回写 Excel；
- 清洗：管理员触发清空岗位和任务数据。

模块实现重点是异常处理和日志记录，避免导入失败时静默出错。

### 6.1.5 AI服务模块实现

`AIService` 对接 GLM-4，包含同步与流式两种模式。

#### （1）同步模式

构建请求体 `model + messages`，发送 HTTP 请求，解析 `choices[0].message.content` 返回。

#### （2）流式模式

设置 `stream=true` 后持续读取 SSE 行，提取 `delta.content` 并推送给前端 `SseEmitter`。  
这种模式可显著改善长文本响应体验，用户无需等待完整结果一次性返回。

#### （3）前置检查与错误兜底

系统在调用前检查 API Key 是否配置；对非 200 响应读取错误流并抛出友好异常，保证调用链可观测。

### 6.2 前端核心实现

### 6.2.1 路由与鉴权

路由集中定义在 `router/index.ts`。`beforeEach` 中读取本地用户状态，未登录自动跳转 `/login`。主布局下挂载首页、看板、岗位、数据管理、分析、设置、AI 页面。

### 6.2.2 API层封装

`http.ts` 实现统一请求配置：

- 请求阶段自动注入 Token；
- 响应阶段统一解析 `code/msg/data`；
- 错误阶段分类提示与登录失效处理；
- 开发环境脱敏日志便于调试。

该封装有效减少页面重复代码并统一错误体验。

### 6.2.3 数据看板页面实现

`Dashboard.vue` 通过并发请求加载统计数据，并构建多种图表：

- 城市分布环形图；
- 薪资区间柱状图；
- 技能需求横向条形图；
- 学历/经验双系列图；
- 热门岗位 TOP 图。

页面对图表加入渐变、动画、悬浮反馈，提升信息可读性与展示效果。

### 6.2.4 岗位列表页面实现

`JobList.vue` 负责查询与业务操作聚合：

- 多条件筛选与分页；
- 岗位卡片展示；
- 一键爬取入口；
- Excel 导出；
- AI分析弹窗、薪资预测弹窗、推荐弹窗。

该页面把“查询—分析—决策”串联为完整交互链路，是系统核心业务入口。

### 6.2.5 数据管理页面实现

`DataManage.vue` 面向管理员，包含任务统计、任务列表、实时日志、数据预览、任务对话框等模块化组件。  
页面内置轮询机制，持续刷新任务状态并更新进度条，保证管理员能直观看到爬取执行过程。

### 6.2.6 AI对话页面实现

`AIChat.vue` 使用组件拆分：`ChatHeader`、`ChatMessageList`、`ChatInput`。  
页面核心能力：

- 支持常用问题快速提问；
- 流式读取 SSE 并逐段渲染；
- 对话状态存入 Pinia；
- 支持重试与清空。

Markdown 渲染使 AI 回答结构化展示，增强可读性。

### 6.3 安全与日志设计

### 6.3.1 安全机制

- 登录认证：JWT 无状态鉴权；
- 接口鉴权：管理员接口使用 `@PreAuthorize`；
- 密码保护：哈希与盐值；
- 错误处理：统一异常响应，避免内部栈信息直接暴露。

### 6.3.2 日志机制

系统在控制器和服务层记录关键行为，内容包含操作类型、任务 ID、执行状态、错误信息。  
日志文件配置在 `logs/recruitment-system.log`，便于定位问题与审计追踪。

### 6.4 关键实现难点与解决方案

#### 难点1：多站点解析兼容性差

解决方案：为主站点分别编写解析函数，同时保留通用解析兜底；选择器采用多候选策略；解析失败可降级备用数据。

#### 难点2：爬取失败导致系统不可用

解决方案：加入重试、随机延时、备用数据、任务状态提示，使系统在网络波动或站点变更时仍保持可演示性。

#### 难点3：数据质量不稳定

解决方案：在写库前进行字段标准化、异常过滤、唯一键去重；对历史数据设置 `OFFLINE` 状态，避免旧数据污染分析。

#### 难点4：AI回答脱离本地数据

解决方案：调用 AI 前注入数据库实时统计上下文，确保回答包含当前数据依据，提高业务相关性。

## 附录A：系统实现补充说明（扩展章节，计入正文）

> 为满足“二稿深化”要求，本附录对若干关键实现进一步展开，作为正文章节的技术补充。

### A.1 动态SQL设计策略

`JobMapper.xml` 中 `selectByCondition` 通过 `<where>` + 多个 `<if>` 组合实现复杂筛选。该方式相较于硬编码 SQL 字符串拼接的优势包括：  
（1）语义清晰，可读性高；  
（2）参数绑定安全，降低注入风险；  
（3）易于后续增删筛选维度；  
（4）与分页插件无缝集成。

在实际使用中，关键词检索会同时覆盖职位名、公司名、技能、职位描述，提升检索召回率。薪资区间条件同时判断 `min_salary` 与 `max_salary`，避免“仅按单字段过滤”造成的数据丢失。

### A.2 任务调度与线程池设计

采集任务采用固定线程池异步执行。这样设计的目的是将长耗时采集从 HTTP 请求线程中解耦，防止接口阻塞。管理员触发“启动任务”后立即得到响应，任务在后台推进。  
任务并发度过高会引发目标站点访问压力、数据库写入冲突和本地资源争抢，因此系统在初版采用保守并发策略，并通过 `SITE_LIMIT` 限制单站点写入数量，保证系统稳定。

### A.3 数据质量治理策略细化

系统中的数据质量治理分为四层：

1. **结构层**：统一字段长度、空值策略、类型约束；  
2. **语义层**：学历、经验、薪资口径统一；  
3. **唯一性层**：基于 `job_key` 去重，避免重复采集污染；  
4. **时效层**：通过 `last_seen_at` 与 `job_status` 维护岗位生命周期。

该策略保证了“可分析数据集”质量，为后续统计与 AI 分析提供可靠输入。

### A.4 可视化表达设计原则

为了让非技术用户也能快速理解数据，图表设计遵循以下原则：

- 图表类型与数据结构匹配（分布用饼图、排名用条形图、比较用柱状图）；  
- 色彩梯度统一，减少认知负担；  
- Tooltip 信息完整但不冗余；  
- 空态可解释，避免“白屏不知所措”；  
- 动画适度，优先保障可读性。

### A.5 AI系统提示词工程实践

`AIController` 中 `SYSTEM_PROMPT` 明确规定了 AI 的角色、能力边界和输出格式，包括 Markdown 层级、列表、数据表格、建议数量等。该设计目的有两点：  
（1）提升回答稳定性，减少风格漂移；  
（2）确保输出可以直接用于前端展示和用户阅读。

在大模型调用中，系统并非仅传入用户问题，而是先注入实时统计上下文，确保回答“有据可依”。这种“提示词 + 数据上下文”组合方案，在工程实践中比单纯聊天更可控。

### A.6 前端状态管理实践

系统将跨页面共享状态集中到 Pinia：用户信息在 `user store`，聊天历史在 `chat store`。  
这种设计可避免多组件重复请求和状态错位问题。尤其在 AI 对话场景中，流式消息需要持续追加，若不集中管理，极易出现组件销毁后状态丢失问题。

### A.7 错误处理分层策略

系统错误处理分三层：

1. **后端业务层**：抛出业务异常并统一转换为 `Result`；  
2. **前端网络层**：Axios 拦截器识别状态码并统一提示；  
3. **页面交互层**：对关键动作给出显式反馈（成功、失败、重试）。

这种分层让错误可定位、可解释、可恢复，显著提升系统可维护性。

### A.8 毕设落地经验总结

在毕业设计实践中，常见误区是过度追求算法复杂度而忽视系统完整性。本文项目实践表明：  
一个可运行、可演示、可复盘的工程系统，比局部“高精尖算法”更具展示价值。  
因此，二稿写作重点应放在“问题—方案—实现—验证—改进”闭环，并结合真实代码说明“为何这样设计、这样设计解决了什么问题”。

## 第9章 工程实现深度复盘（扩展章节）

> 本章为二稿扩展内容，重点回答“系统为什么这样实现、实现过程如何权衡、实际运行中遇到了什么问题、如何持续优化”，用于提升论文深度与字数完整性。

### 9.1 从“功能可用”到“工程可用”的转变

很多毕业设计在中期阶段都能完成“功能演示”，例如：可以登录、可以查数据、可以画图、可以调用 AI。但当系统进入连续运行、多人测试、边界输入、异常场景后，常见问题迅速暴露：页面偶发空白、任务状态不一致、导出文件失败、AI 响应偶发超时、数据统计口径前后不一致。  
这些问题说明：系统仅满足“功能可用”还不够，需要向“工程可用”升级。

本系统在二稿阶段围绕工程可用性做了如下加强：

1. **统一返回结构**：后端统一 `Result` 包装，前端统一拦截处理，避免每个页面重复写异常分支；  
2. **统一鉴权策略**：`SecurityConfig + JwtAuthFilter` 形成完整鉴权链路，避免权限逻辑散落；  
3. **统一日志风格**：关键路径带上任务ID、用户ID、模块名，提升定位速度；  
4. **统一任务状态机**：`PENDING/RUNNING/FINISHED/FAILED` 明确任务生命周期，前端可据此稳定渲染；  
5. **统一降级策略**：爬虫失败可启用备用数据，AI失败有错误提示和规则兜底，系统不至于“不可演示”。

### 9.2 架构边界与模块职责再梳理

#### 9.2.1 控制器职责边界

控制器层只做三件事：参数接收、基础校验、调用服务。  
在早期版本中，部分数据处理逻辑曾放在 Controller，导致接口臃肿、难以测试。重构后将复杂逻辑移入 Service，使控制器保持“薄层”。

#### 9.2.2 服务层职责边界

服务层负责业务规则与流程编排。例如：

- `CrawlServiceImpl`：负责任务执行全链路；
- `JobServiceImpl`：负责统计、预测、推荐；
- `DataServiceImpl`：负责导入导出与清洗；
- `UserServiceImpl`：负责认证相关业务规则。

这样的边界让后续扩展更容易。例如未来增加新站点时，主要改动集中在 `CrawlServiceImpl` 与相关映射层，不会影响前端与认证模块。

#### 9.2.3 数据访问层职责边界

Mapper 层只处理数据映射，不承载业务流程。  
复杂统计尽量在 SQL 层完成，避免大量原始数据回传后端再聚合，这样更节省网络与内存开销。

### 9.3 数据采集模块深度分析

### 9.3.1 任务模型的必要性

若仅提供“立即采集”按钮，系统很快会遇到问题：用户刷新页面后不知道任务是否还在执行；失败后无法追溯；并发触发会互相覆盖。  
引入任务模型后，采集行为具备了可追踪性和可管理性。任务对象记录了来源站点、关键词、城市、状态、数量、消息、创建时间、结束时间。前端可以基于任务模型实现管理面板和进度反馈。

### 9.3.2 采集稳定性的工程策略

采集稳定性不是单点技巧，而是策略组合：

1. **请求层**：设置超时、重试、延迟、随机 UA；  
2. **解析层**：多选择器、通用兜底、字段清洗；  
3. **存储层**：去重、更新、状态修正；  
4. **监控层**：任务日志和状态统计。

这些策略组合起来，才能在网络波动、页面变更、字段缺失时保持“系统仍可运行”。

### 9.3.3 降级数据的边界与意义

在教学项目中，完全依赖真实站点数据有一定风险：网络限制、访问限制、页面突变都可能导致“当场无数据可展示”。因此系统提供了备用数据生成机制。  
需要强调的是，备用数据的目标是保证流程可验证，不是替代真实数据。论文中应清楚说明：备用数据用于系统可演示性与容错，不用于夸大分析结论。

### 9.4 数据治理模块深度分析

### 9.4.1 为什么要做字段标准化

招聘数据来自多平台，字段表达高度不统一，例如薪资可能是“15K-25K”“1.5-2.5万”“面议”；经验可能是“1-3年”“三年以上”“经验不限”。  
若不做标准化，统计分析会失真，图表结果不可比。系统通过规则映射将同义表达归并到统一口径，为后续统计提供基础。

### 9.4.2 去重策略的实际价值

同一岗位可能在多个时间点被重复采集，若不去重会导致：

- 岗位数量被虚高；
- 薪资均值被重复样本影响；
- 推荐结果重复岗位过多。

系统使用 `job_key` 作为唯一识别字段，实现“存在则更新，不存在则新增”，并配合 `last_seen_at` 标记岗位活跃状态。

### 9.4.3 状态治理策略

系统采用 `NEW/ACTIVE/OFFLINE` 三区分，不直接删除历史岗位。  
这样做有三点好处：  
（1）可追踪岗位生命周期；  
（2）可做下线趋势分析；  
（3）避免误删导致数据断层。

### 9.5 统计分析模块深度分析

### 9.5.1 统计口径一致性

统计最怕“同名指标不同口径”。本系统在 SQL 层统一口径：  
默认统计排除 `OFFLINE`，避免历史失效岗位干扰当前分析。  
同一指标在看板和分析页复用同源接口，保证口径一致。

### 9.5.2 技能统计的复杂性

`skills` 为逗号分隔字符串，不是规范化子表。当前实现通过 SQL 拆分实现统计，优点是实现快，缺点是规模扩大后性能一般。  
未来建议将技能拆成 `job_skill` 关联表，并在写入阶段完成标准化映射，这样统计和推荐都会更稳定。

### 9.5.3 薪资分析的可解释性

本系统薪资统计使用 `(min_salary + max_salary) / 2` 作为岗位均值，有助于跨岗位比较。  
对于“面议”岗位则在清洗阶段避免无效值污染。  
虽然该方法不是严格经济学建模，但在工程实践中直观、可解释、可实现。

### 9.6 AI模块深度分析

### 9.6.1 AI不是替代统计，而是增强入口

系统中的 AI 功能定位并不是替代传统统计，而是将统计结果“语言化、建议化”。  
用户可以通过自然语言快速获得“结论 + 建议”，降低使用门槛。

### 9.6.2 上下文增强的重要性

如果仅把用户问题直接发给模型，回答很容易泛化。系统通过 `buildEnhancedMessage` 注入实时统计上下文（热门城市、技能、薪资分布、趋势、热门岗位）后，回答更贴近本地数据。这是本系统 AI 模块最关键的工程价值点。

### 9.6.3 流式输出的用户体验价值

同步返回在长文本场景中等待感强。SSE 流式推送将“等待整段完成”变成“边生成边阅读”，用户主观体验提升明显。  
在前端实现上，系统将每个 chunk 追加到同一 AI 消息，并按节奏渲染，避免频繁重排造成卡顿。

### 9.6.4 AI异常场景处理

AI模块必须考虑异常：

- 未配置 API Key；
- 外部接口超时；
- 返回格式异常；
- 网络中断。

系统在后端和前端分别做了提示与兜底，确保异常可见、可理解、可恢复。

### 9.7 前端工程实践复盘

### 9.7.1 组件化拆分收益

以数据管理页为例，拆分为 `DataManageHeader`、`StatCards`、`CrawlProgress`、`TaskTable`、`DataPreviewTable` 等组件后，页面主文件主要负责状态编排，结构清晰、可维护性明显提升。

### 9.7.2 状态管理实践收益

`user store` 解决跨页面登录态一致性；  
`chat store` 解决 AI 对话持久化与流式追加。  
若不使用全局状态，页面跳转后消息丢失、重复请求和状态错乱问题会频繁出现。

### 9.7.3 统一错误体验

通过 Axios 拦截器统一处理错误提示，让用户在不同页面遇到网络或权限问题时获得一致反馈，减少“有的页面提示、有的页面静默失败”的体验割裂。

### 9.8 安全设计复盘

### 9.8.1 鉴权链路完整性

后端默认“除白名单外全部鉴权”，可避免遗漏受保护接口。  
管理员能力通过 `@PreAuthorize` 限制，确保普通用户不能执行高风险操作（如清库、任务管理、删除岗位）。

### 9.8.2 密码与Token安全

密码使用哈希盐值，不落明文；Token 设置有效期并在前端过期后清理本地状态。  
虽然课程项目未接入更高级的安全机制（如刷新令牌、设备管理、风控策略），但当前实现已达到教学系统的基础安全要求。

### 9.9 测试复盘与质量保障

### 9.9.1 测试策略分层

本系统采用“接口功能测试 + 页面交互测试 + 异常场景测试 + 基础性能测试”的组合方式。  
其中异常场景测试尤为关键，因为系统核心价值之一是“出错时仍能提供可用反馈”。

### 9.9.2 典型缺陷复盘

在开发中出现过以下典型问题：

1. **任务状态迟滞**：前端轮询间隔过长导致状态更新不及时；  
2. **图表空态崩溃**：接口返回空数组时图表配置未判空；  
3. **Token失效后循环跳转**：拦截器与路由守卫处理顺序冲突；  
4. **导出Blob处理不一致**：响应拦截器与页面代码对 blob 的处理方式不统一。

针对这些问题，系统逐步引入判空、兜底、统一拦截策略和明确的状态机约束，稳定性显著提升。

### 9.10 代码可维护性策略

### 9.10.1 命名与目录结构

后端按 `controller/service/mapper/entity/dto/vo/config/util` 组织，前端按 `views/components/api/store/router` 组织。  
这种结构对团队协作和后期维护非常友好。

### 9.10.2 配置外置化

数据库账号、JWT密钥、AI密钥等通过配置文件和环境变量管理，避免硬编码，提高部署灵活性与安全性。

### 9.10.3 低耦合扩展设计

例如新增招聘站点时，主要在爬虫模块新增 URL 构建与解析逻辑；统计与可视化模块可复用原有数据接口。这体现了模块低耦合的价值。

### 9.11 生产化改造建议（中长期）

若系统要走向真实生产环境，建议按以下路径升级：

1. 将爬虫执行从本地线程池迁移到消息队列 + 分布式任务调度；  
2. 对高频统计结果增加缓存层（Redis）；  
3. 引入指标监控（响应时间、错误率、任务成功率）；  
4. 增加细粒度权限模型与审计追踪；  
5. 为 AI 调用增加熔断、重试、限流与计费监控；  
6. 引入自动化测试流水线，提高迭代质量。

### 9.12 本章小结

本章从工程视角对系统进行复盘，强调了“系统价值不只在于能跑起来，更在于能稳定运行、可追踪、可扩展、可解释”。  
通过任务模型、数据治理、统计口径统一、AI上下文增强、统一错误处理与权限设计，系统形成了较完整的工程闭环。  
这些实践经验同样适用于其他“数据密集+智能服务”类管理系统开发。

## 第10章 论文写作反思与方法论沉淀（扩展章节）

> 为满足二稿深度要求，本章总结本课题在“技术实现”之外的方法论收获，帮助后续终稿进一步学术化。

### 10.1 问题驱动而非技术堆砌

在项目初期，最容易出现“先选一堆技术，再找应用场景”的倾向。  
本课题通过反向路径改进：先明确招聘场景中的真实痛点，再对应技术手段。  
例如：

- 痛点：多平台数据碎片化  
  技术：爬虫任务模型 + 字段标准化  
- 痛点：结果难理解  
  技术：可视化看板 + AI自然语言解释  
- 痛点：系统不稳定  
  技术：重试、降级、日志、状态机

这种“问题-方案”映射关系，是论文论证的核心逻辑。

### 10.2 “可解释性”优先于“复杂度”

在毕业设计语境下，过于复杂的模型如果无法解释、无法复现，反而削弱论文质量。  
本课题在薪资预测和岗位推荐中优先采用规则与统计方法，虽然算法上不“炫”，但具备以下优点：

1. 计算过程透明，便于论文表达；  
2. 结果可解释，便于用户理解；  
3. 实现成本低，便于在项目周期内完成；  
4. 后续可平滑升级到机器学习模型。

### 10.3 工程论文的“证据链”写法

优质工程论文应建立证据链：

1. **需求证据**：问题真实存在；  
2. **设计证据**：架构选择有依据；  
3. **实现证据**：关键模块可追踪；  
4. **测试证据**：结论有数据支持；  
5. **反思证据**：明确不足与改进路径。

本二稿已尽量按照这条链路组织章节，从而避免“只讲概念、不讲落地”。

### 10.4 如何把代码实现转化为论文语言

代码是实现细节，论文是结构化论证。转换时需要注意：

- 不要逐行解释代码，而是解释设计意图与关键机制；  
- 不要只写“实现了什么”，还要写“为什么这样实现”；  
- 不要只写成功结果，还要写失败场景与兜底策略；  
- 不要只写功能列表，还要写模块协同关系。

例如，爬虫模块不应只写“实现了多平台采集”，而应写“为什么需要任务状态机、为什么需要重试和降级、这些机制如何提高可用性”。

### 10.5 面向答辩的内容强化建议

终稿前建议围绕答辩高频问题补强材料：

1. **你系统的创新点是什么？**  
   回答重点：工程闭环与AI数据上下文增强。  
2. **为什么不用更复杂的模型？**  
   回答重点：可解释、可实现、可迭代。

## 第11章 关键模块逐项技术说明（扩展章节）

### 11.1 登录与会话治理模块

系统登录模块看似简单，实际上承担“系统安全边界入口”的关键责任。项目采用 `用户名+密码` 的基础认证模型，登录成功后由后端生成 JWT，并在前端本地持久化。相比传统 Session 方案，JWT 更适合前后端分离：服务端无需保存会话状态，系统可水平扩展，接口调用链更简单。  
然而 JWT 也带来新问题，如 Token 过期、失效处理、客户端状态一致性等。系统在 `http.ts` 的响应拦截中对 `401` 统一处理：提示“登录已过期”，清理本地用户状态并跳转登录页，保证异常路径可控。  
此外，系统通过 `beforeEach` 守卫在路由进入前读取本地用户状态，解决浏览器刷新后内存状态丢失导致误判的问题。这种“拦截器 + 路由守卫 + 本地状态同步”的组合，是保证登录体验连续性的关键。

### 11.2 权限控制模块

权限控制由两层组成：  
第一层是认证层（是否登录）；  
第二层是授权层（是否具备操作权限）。

系统在后端使用 Spring Security 进行统一拦截，默认除白名单外全部鉴权。管理员接口增加 `@PreAuthorize`，例如爬虫管理、数据清洗、岗位写操作等。  
`JwtAuthFilter` 根据角色动态注入权限列表：管理员拥有 `job:write`、`job:delete`、`crawl:manage`、`data:cleanup` 等权限，普通用户仅保留基础访问能力。这样做可以在不改业务代码的情况下统一控制访问边界，减少权限漏洞风险。  
在论文论证中，这体现了“最小权限原则”的工程实践：每个角色只授予完成业务所需的最小集合权限。

### 11.3 数据采集解析模块

采集模块由“任务管理 + 请求执行 + DOM解析 + 数据落库”四部分构成。  
任务管理负责定义采集范围，请求执行负责稳定抓取，DOM解析负责结构抽取，数据落库负责统一存储。  
系统支持 BOSS、智联、前程无忧、猎聘、拉勾、牛客等多个来源，URL 构建逻辑按站点分支处理，避免统一模板导致参数错误。

在解析阶段，系统通过多组 CSS 选择器提取岗位信息。由于站点页面结构经常调整，单选择器策略稳定性不足，因此系统采用“多候选选择器 + 通用兜底解析”。  
在字段质量方面，系统对标题长度、公司名有效性、岗位文本完整度进行过滤，避免垃圾数据进入数据库。  
在业务适配方面，系统加入了高级岗位过滤、无效岗位过滤、薪资阈值过滤、学历过滤等策略，使数据集更贴近“应届与初中级招聘分析”场景。

### 11.4 岗位去重与生命周期治理模块

招聘数据天然存在重复采集问题。系统以 `source + title + company + city` 组合生成哈希 `job_key` 作为唯一标识，实现幂等写入。  
写入逻辑分为两类：  
（1）新岗位：写入并标记 `NEW`；  
（2）已存在岗位：更新关键字段并标记 `ACTIVE`，同步刷新 `last_seen_at`。

同时，系统在每轮采集后对“未再次出现”的岗位进行下架标记（`OFFLINE`），形成岗位生命周期闭环。  
这种机制比“直接删除旧岗位”更科学，因为它保留了历史信息，可用于趋势分析和下线统计。  
此外，系统引入了最近岗位抽样验证机制，避免全量验证带来的网络压力。抽样结果主要用于风险预警，而不是直接大规模状态切换，降低误判风险。

### 11.5 薪资解析与标准化模块

薪资是招聘分析中最关键但最难标准化的字段之一。不同平台表达方式差异巨大，如 `10K-15K`、`1-1.5万`、`面议` 等。  
系统通过正则匹配提取区间数值并统一换算为月薪数值，无法解析或面议情况保留为空，避免错误估算。  
在统计层，系统使用 `(min_salary + max_salary)/2` 作为岗位薪资近似值参与聚合。该近似虽非完美，但在大样本下能有效反映总体薪资结构，且计算成本低、解释性强。  
在预测层，系统采用“样本均值 + 因子修正”策略，因子来自城市、经验、学历、技能溢价。相较于黑盒模型，这种方式更适合课程论文论证，因为每个调整项都可解释其业务含义。

### 11.6 技能标签抽取模块

技能标签抽取采用词库匹配方案：系统内置多类别技能词表（后端、前端、数据AI、运维、测试等），并使用“长词优先”避免子串重复。  
例如已命中 `SpringBoot` 时不再重复计入 `Spring`，提高标签质量。  
标签提取上限设置为固定数量，防止单岗位标签过长影响存储与展示。  
该方案优势是实现成本低、可解释性强、可随业务迭代词库；不足是无法识别复杂语义和隐式技能。后续可引入分词与语义模型增强。

### 11.7 统计查询与图表联动模块

统计模块以 SQL 聚合为核心，避免将大量原始数据拉回后端再处理。  
前端看板通过并发接口请求快速获取多维数据，再根据图表类型映射为 ECharts option。  
为提升阅读体验，系统对图表样式进行了统一设计，包括：渐变色体系、卡片层级、tooltip 格式、label 可读性、动画节奏等。  
图表并非“越炫越好”，而是要保证用户一眼看懂：哪里岗位多、哪里薪资高、哪些技能热、哪类学历经验最常见。

### 11.8 AI智能分析模块

AI模块由两条路径组成：

1. **问答路径**：用户自然语言提问，系统拼接数据库上下文并调用模型；  
2. **分析路径**：系统先规则聚合，再输出结构化分析对象给前端展示。

这种双路径设计的意义在于平衡灵活性与稳定性。  
问答路径灵活，但可能受模型波动影响；分析路径结构稳定，可作为兜底。  
在流式交互层，系统采用 SSE 将模型输出逐段发送，前端实时渲染，显著提升体验。  
在异常场景下，若 API Key 缺失或模型接口失败，系统返回明确错误提示，避免用户误以为页面卡死。

### 11.9 数据导入导出模块

数据导入导出是管理系统常见但容易忽视的模块。  
导入阶段需要处理空行、格式错误、日期解析失败等问题，系统在服务层做了防御性校验。  
导出阶段需考虑浏览器兼容、文件名编码、响应类型处理，系统通过 blob 下载流程统一实现。  
该模块既方便课程演示，也便于后续与外部数据源对接。

### 11.10 日志与可观测性模块

系统日志不仅用于排错，还用于“解释系统行为”。  
例如任务执行日志可回答：何时开始、何时结束、为何失败、新增多少、更新多少、下架多少。  
用户行为日志可回答：谁在什么时间执行了何种敏感操作。  
这些信息在毕业设计答辩中同样有价值，因为它体现了系统从“功能工具”向“可管理系统”的提升。

### 11.11 前端交互体验模块

系统在交互设计上重点关注三类体验：

1. **反馈即时性**：按钮操作后有明确 loading、成功、失败提示；  
2. **信息可读性**：卡片、标签、分页、空态、图表均保持一致风格；  
3. **连续性**：聊天记录、登录态、筛选条件在合理范围内保持连续。

这些设计减少了用户学习成本，也降低了误操作概率。  
在论文写作中，交互体验不应被视为“UI装饰”，而应作为系统可用性的重要组成部分。

### 11.12 异常处理闭环模块

系统异常处理遵循“可见、可判、可恢复”三原则：

- 可见：错误信息必须反馈给用户或日志；
- 可判：错误类型可区分（权限、网络、服务、业务）；
- 可恢复：给出重试、重新登录、检查配置等下一步动作。

例如 AI 流式失败时，系统会在聊天区域写入错误提示而不是沉默失败；导出失败时提示用户重试；权限不足时返回 403 并明确说明。  
这种设计提升了系统可靠感，也降低了维护人员排障成本。

### 11.13 本章小结

本章逐项说明了系统关键模块的设计意图、实现方式与工程价值。  
通过这些细化分析可以看出：本系统不是若干独立功能拼接，而是围绕招聘数据闭环构建的协同系统。  
这种模块化协同思路，是后续持续演进和生产化改造的基础。

## 第12章 部署运行与运维实践（扩展章节）

### 12.1 本地部署流程说明

为保证系统可复现，本项目将部署流程拆分为数据库初始化、后端启动、前端启动三步。

### 12.2 配置管理实践

系统配置集中在 `application.yml`，包括：

- 服务端口与编码；
- 数据库连接与连接池参数；
- MyBatis 与分页插件配置；
- 日志输出与文件路径；
- JWT 密钥和有效期；
- AI 接口密钥读取；

将关键配置参数化有三大收益：  
（1）便于环境切换（开发、测试、演示）；  
（2）便于快速定位故障（通过日志级别与连接配置）；  
（3）便于后续迁移（例如从本地 MySQL 迁移到云数据库）。

### 12.3 运行监控与问题排查流程

在缺少完整可观测平台的前提下，系统采用“日志 + 状态页 + 管理面板”的轻量监控方式：

1. 管理员在任务列表查看任务状态与执行消息；  
2. 在数据管理页查看实时日志和进度；  
3. 在后端日志文件中定位具体异常栈；  
4. 在前端控制台查看接口请求与错误提示。

这种多层排查路径可覆盖大多数课程项目问题。  
例如“任务一直 RUNNING 不结束”时，可先看任务消息，再看后端日志定位网络超时或解析失败；“页面无数据”时，可先查接口响应，再查数据库表数据是否写入成功。

### 12.4 常见故障与处理手册

#### 故障1：后端启动失败

常见原因：数据库未启动、账号密码错误、端口冲突、JDK版本不匹配。  
处理步骤：检查 MySQL 服务、检查配置、查看启动日志中的首个错误栈、确认 JDK 版本与编译目标一致。

#### 故障2：前端请求 401

常见原因：Token 过期、本地用户信息损坏、未登录直接访问受限页面。  
处理步骤：清理本地存储重新登录；检查拦截器是否正确附加 Bearer Token。

#### 故障3：爬虫任务大量失败

常见原因：目标站点访问限制、网络波动、页面结构变化。  
处理步骤：先确认网络可达，再查看重试日志；必要时开启备用数据保障流程；后续更新解析选择器。

#### 故障4：AI 无返回

常见原因：未配置 `ZHIPUAI_API_KEY`、外部接口超时。  
处理步骤：检查环境变量和后端提示；通过同步模式验证接口可达；必要时启用规则报告兜底。

#### 故障5：导出文件损坏

常见原因：前端把 blob 当 JSON 解析、响应头设置错误。  
处理步骤：确认 `responseType='blob'`，并使用对象 URL 下载。

### 12.5 数据备份与恢复建议

尽管课程项目规模较小，数据备份仍很有必要。建议采用：

1. 每日导出数据库快照；  
2. 关键演示前手动备份；  
3. 清洗操作前二次确认并记录操作人；  
4. 保留一份稳定演示数据集，避免临时采集失败影响答辩。

这类运维习惯能显著提升项目稳定性，也体现了工程意识。

### 12.6 性能优化路线图

系统未来可按“低成本优先”原则逐步优化：

第一阶段：SQL索引优化、分页上限控制、图表懒加载；

可视化图表帮助用户快速识别热门城市、热门技能和主流薪资区间；AI 问答则将复杂统计转化为可执行建议，例如“某技能是否值得学习”“某城市岗位是否适合应届生”“当前经验阶段如何提升竞争力”。  
相比纯列表检索，这种“数据+解释+建议”的方式更贴近求职者真实需求。

### 13.3 教学与科研场景价值

在教学场景中，本系统可作为完整案例覆盖软件工程多个环节：需求分析、架构设计、数据库建模、接口开发、前端可视化、鉴权安全、日志治理、AI 集成。  
在科研场景中，系统可作为实验平台，用于验证不同推荐策略、薪资预测模型、技能抽取方法的效果。  
因此，本系统不仅是一个毕业项目成果，也可作为后续课程实践与团队协作开发的基础平台。

# AI招聘数据可视化系统的设计与实现（二稿）

## 摘要

随着数字经济发展与人工智能技术加速落地，招聘行业正在从“信息发布”向“数据驱动决策”转变。BOSS直聘、智联招聘、前程无忧等平台每天产生海量岗位数据，但现实中仍存在数据分散、字段不统一、分析维度单一、决策支持不足等问题，导致企业招聘效率不高、求职者决策成本较大。针对上述痛点，本文设计并实现了一个面向招聘场景的 AI 招聘数据可视化系统。

系统采用前后端分离架构，以“数据采集—数据清洗—数据存储—可视化分析—AI智能服务”为核心流程。后端基于 Java、Spring Boot 与 MyBatis 实现业务服务与数据访问，前端基于 Vue 3、TypeScript、Element Plus 与 ECharts 构建交互式可视化界面，数据库采用 MySQL。系统通过爬虫组件实现多平台岗位数据自动采集，利用规则化清洗方法对薪资、学历、经验、技能标签等字段进行标准化处理，并集成大模型接口实现智能问答、岗位推荐与薪资预测功能。

实验与测试结果表明：系统在功能完整性、交互体验、数据处理准确性和运行稳定性方面均达到预期目标。该系统可为企业招聘策略优化与求职者职业决策提供有效支持，对人力资源数字化与智能化建设具有一定实践价值与推广意义。

**关键词**：招聘数据；数据可视化；前后端分离；Spring Boot；Vue 3；爬虫技术；人工智能

## Abstract

With the rapid development of digital economy and artificial intelligence, the recruitment industry is shifting from simple information publishing to data-driven decision-making. Mainstream recruitment platforms generate massive job data every day, while practical scenarios still suffer from fragmented data, inconsistent field formats, limited analytical dimensions, and weak decision support. To address these issues, this paper designs and implements an AI-based recruitment data visualization system.

The system adopts a front-end/back-end separated architecture and follows a complete pipeline of data collection, data cleaning, data storage, visual analysis, and AI services. The back-end is implemented with Java, Spring Boot, and MyBatis; the front-end is developed using Vue 3, TypeScript, Element Plus, and ECharts; MySQL is used for persistent storage. The system enables multi-platform job data crawling, standardized data cleaning for salary/education/experience/skill fields, and intelligent functions such as AI Q&A, job recommendation, and salary prediction through large-model integration.

Test results show that the system meets expected requirements in terms of functional completeness, interaction quality, data processing accuracy, and operational stability. It can provide practical support for enterprise recruitment optimization and job seekers’ career decisions, and has clear application value in HR digital transformation.

**Key Words**: Recruitment Data; Data Visualization; Front-end/Back-end Separation; Spring Boot; Vue 3; Crawler; AI Service

## 第1章 绪论

### 1.1 研究背景

近年来，互联网招聘平台快速发展，岗位信息规模持续增长。传统招聘分析方式主要依赖人工统计与经验判断，难以满足“高频更新、跨平台整合、实时分析”的业务需求。企业在招聘中普遍面临以下问题：  
- 岗位数据来源多、格式不统一，数据治理成本高；  
- 招聘策略缺乏量化支撑，岗位投放与薪资策略优化困难；  
- 人岗匹配效率低，候选人筛选周期较长。  

同时，求职者也面临信息噪声大、岗位比较成本高、薪资预期不明确等问题。构建一个集数据采集、清洗、可视化与智能分析于一体的系统，具有明显的现实意义。

### 1.2 国内外研究现状

国外招聘平台在数据治理与智能推荐方面起步较早，普遍采用多维度行为数据建模、人岗匹配算法与预测分析技术。国内平台在岗位检索、简历管理等功能上已较成熟，但在“多平台数据融合”“高质量数据标准化”“可解释可视化分析”及“统一AI服务入口”方面仍有较大提升空间。

从技术趋势看，前后端分离架构、可视化分析引擎、流式 AI 交互已成为招聘系统升级方向。本文在工程实现上将上述能力进行一体化整合，形成面向实际应用的技术方案。

### 1.3 研究目标与内容

本文围绕“可采、可管、可视、可智”四个目标展开：

1. **可采**：实现多平台招聘数据自动化采集与任务管理。  
2. **可管**：实现招聘数据清洗、去重、标准化与检索管理。  
3. **可视**：提供城市分布、薪资结构、技能热度等多维可视化分析。  
4. **可智**：引入 AI 服务，支持招聘问答、岗位推荐与薪资预测。  

### 1.4 研究方法与技术路线

本文采用“需求分析—总体设计—模块实现—系统测试—总结优化”的研究路径。技术路线如下：

- 使用爬虫实现招聘数据获取；  
- 使用规则化清洗与结构化存储保证数据质量；  
- 通过可视化图表构建分析视图；  
- 通过大模型接口提供自然语言智能服务。  

### 1.5 论文结构安排

全文共分为八章：绪论、相关技术、可行性研究、系统分析、系统设计、系统实现、系统测试、结论与展望。

## 第5章 系统总体设计

### 5.1 架构设计

系统采用前后端分离与分层架构，整体分为：

1. 前端展示层（UI、交互、图表）；  
2. 业务服务层（认证、任务调度、数据治理、AI服务）；  
3. 数据访问层（Mapper/DAO）；  
4. 数据存储层（MySQL）。  

该架构实现了职责分离，便于独立演进和持续优化。

### 5.2 核心模块设计

- **数据采集模块**：任务调度、站点解析、反爬策略与失败重试；  
- **数据治理模块**：字段标准化、去重、空值处理、异常过滤；  
- **数据分析模块**：统计聚合、条件筛选、结果缓存；  
- **可视化模块**：图表渲染、交互筛选、导出下载；  
- **AI服务模块**：流式问答、推荐计算、预测分析；  
- **系统管理模块**：用户角色、权限校验、行为日志。  

### 5.3 数据库设计

核心表包括：

- `user`：用户信息与角色权限；  
- `company`：企业基础信息；  
- `job`：岗位核心数据（薪资、学历、经验、技能等）；  
- `crawl_task`：采集任务配置与执行状态；  
- `sys_log`：关键操作与运行日志。  

设计重点在于：主键与唯一约束保证一致性，索引优化提升查询性能，字段规范化提高分析质量。

## 第6章 系统详细设计与实现

### 6.1 数据采集模块实现

系统通过任务驱动方式执行采集：管理员配置平台、关键词、城市后触发采集任务；后端根据站点解析器抓取页面并抽取字段。为提升稳定性，采用 UA 轮换、超时重试、频率限制等策略。

### 6.2 数据清洗与管理实现

针对原始招聘数据中“格式混乱、语义不一致、重复数据多”等问题，本文重点实现：

- 薪资区间解析（如 `15K-25K` 统一为数值区间）；  
- 学历与经验口径统一（如“本科及以上”归一到标准标签）；  
- 基于岗位关键字段哈希去重；  
- 空值与异常值过滤。  

管理端支持分页检索、条件筛选、批量导入导出与人工修正，形成“自动治理 + 人工兜底”的数据质量保障机制。

### 6.3 可视化分析模块实现

系统通过聚合接口输出统计结果，前端使用 ECharts 实现图表展示。核心分析视图包括：

- 岗位城市分布；  
- 薪资区间分布；  
- 技能关键词热度；  
- 行业招聘热度趋势；  
- 学历与经验需求结构。  

图表支持联动筛选与结果导出，满足多维分析需求。

### 6.4 AI智能服务模块实现

1. **AI问答**：前端提交自然语言问题，后端调用大模型并通过 SSE 流式返回，提升交互实时性；  
2. **岗位推荐**：根据技能、学历、经验、城市进行加权匹配并返回推荐列表；  
3. **薪资预测**：结合历史岗位样本与条件特征输出薪资区间参考值。  

该模块实现了从“数据展示”到“智能决策辅助”的能力升级。

### 6.5 系统管理模块实现

系统实现基于角色的访问控制（RBAC），对关键操作进行审计记录。管理员可进行用户状态管理、权限分配、日志检索与问题追踪。
